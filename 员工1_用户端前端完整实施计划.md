## 2. 员工 1 的角色定位

员工 1 是唯一的统一前端负责人，不再只负责普通用户页面。

### 2.1 普通用户工作区

员工 1 负责：

### 2.2 管理员管理中心

- 文档上传、预览、状态、重试、删除和下载。
- 文档转换任务。
- 命中率测试。

### 2.3 工程质量

员工 1 负责：

- `frontend/web/` 工程骨架。
- Vue Router、Pinia 和统一 API Client。
- OpenAPI 生成类型接入。
- 类型安全 Mock。

## 4. 统一前端架构约束

### 4.1 唯一前端工程

```text
frontend/
└── web/
    ├── src/
    │   ├── api/
    │   ├── assets/
    │   ├── components/
    │   ├── composables/
    │   ├── features/
    │   │   ├── auth/
    │   │   ├── chat/
    │   │   ├── conversations/
    │   │   ├── documents/
    │   │   ├── exports/
    │   │   ├── retrieval/
    │   │   └── admin/
    │   ├── layouts/
    │   ├── mocks/
    │   ├── router/
    │   ├── stores/
    │   ├── styles/
    │   ├── tests/
    │   ├── types/
    │   └── views/
    ├── README.md
    ├── package.json
    ├── tsconfig.json
    └── vite.config.ts
```

禁止重新建立：

```text
frontend/user-web/
frontend/admin-web/
frontend/shared-ui/
```

### 4.2 布局

```text
App
├── LoginLayout
└── AuthenticatedLayout
    ├── UserWorkspaceLayout
    └── AdminWorkspaceLayout
```

- 所有账号使用同一个登录入口。
- 管理员默认仍可进入普通用户工作区。
- 当前账号具有管理查看权限时，顶部显示“管理中心”。
- 管理中心使用 `/admin/*` 路由组。
- 普通用户手工访问 `/admin` 时显示 403，且不预加载管理数据。

### 4.3 权限

- 不通过用户名、邮箱或固定角色名判断管理员。
- `GET /api/v1/me` 返回角色、权限码和知识库数据范围。
- 路由使用权限元数据。
- 按钮使用统一 `usePermission().can(permission)`。
- 前端隐藏只能减少误操作，后端必须最终鉴权。
- 权限撤销后退出失权页面，不保留旧管理数据。

### 4.4 状态管理

Pinia 只保存：

- 当前用户和权限。
- 当前知识库。
- 当前会话的必要摘要。
- 真正跨页面的共享状态。

以下状态保留在页面或 feature：

- 表单。
- 弹窗。
- 页码和排序。
- 单页筛选。
- 上传临时列表。
- 单次导出配置。
- 页面加载和错误。

## 6. Issue 和分支规则

### 6.1 总跟踪 Issue

先创建一个不对应代码分支的 Issue：

```text
员工 1｜统一前端 P0 完整交付
```

- 后端依赖。
- 阻塞项。
- PR 链接。
- P0 验收结果。

### 6.2 功能 Issue 必填内容

- 用户问题和价值。
- 本次目标。
- 包含和不包含范围。
- 页面和交互状态。
- API 和权限依赖。
- 验收标准。
- 本地测试方法。
- 失败和边界场景。
- 分支名称。
- PR 链接。

### 6.3 分支规则

- GitHub 创建 Issue 后，用实际编号替换 `<issue>`。
- 不提前创建全部分支。
- 每次只创建即将开发的分支。
- 每个新分支从最新 `main` 创建。
- 依赖上一个分支时，等待其合并。
- 一个 Issue 对应一个分支和一个 PR。

## 11. 风险和处理

| 风险 | 处理 |
|---|---|
| 员工 1 单人承担全部前端 | P0 优先、小分支、README、Mock 和测试；延误时先削减 P1 |
| OpenAPI 延迟 | 先使用生成类型对应 Mock；真实契约未通过不关闭 Issue |
| 用户和管理组件互相污染 | 业务组件留在 feature，真正跨模块复用后再上移 |
| 权限显示和后端不一致 | 权限码单一来源、后端兜底、权限矩阵测试 |
| Markdown XSS | 服务端清洗加 DOMPurify 二次过滤 |
| 模型密钥泄露 | 只写、不回显、不持久化、提交后清空 |
| SSE 断流 | 增量缓冲、取消、半包和断流测试 |
| 上传和导出轮询泄漏 | 终态、失败或页面离开时停止 |
| PR 评审不及时 | 分支保持小规模，提前指定评审人 |
| P1 插入首期 | 新需求默认进入第二阶段 Issue |

## 0.5 治理背景与本计划的交付模式

### 0.5.1 治理背景（从 6 人版到 5 人版）

原《知识库平台六人团队分工协作方案》文档版本 2.0，团队规模 6 人，前端拆为用户端与管理员端两个独立工程。治理调整后，团队规模收缩为 5 人（员工 1、3、4、5、6），前端合并为统一工程。

与原方案相比，治理变化如下：

| 维度 | 6 人版（基线） | 5 人版（本计划） |
|---|---|---|
| 前端工程 | `frontend/user-web` + `frontend/admin-web` + `frontend/shared-ui` | `frontend/web/`（唯一前端工程） |
| 前端负责人 | 员工 1（用户端）+ 员工 2（管理端） | 员工 1（统一前端） |
| 公共组件 | `frontend/shared-ui` 独立维护 | 取消 `shared-ui`，公共组件直接放在 `frontend/web/src/components/`，归属按 feature 划分 |
| 部署容器 | `user-web`、`admin-web`、`api-server`、`worker`、`postgres`、`redis`（6 个） | `web`（Nginx 托管 SPA）、`api-server`、`worker`、`postgres`、`redis`（5 个） |
| 路由 | 登录入口分离 | 统一登录入口，管理员额外看到 `/admin/*` |
| 接口范围 | 员工 1 只对接用户端 | 员工 1 同时对接用户端与管理中心接口 |

### 0.5.3 本计划的交付模式（与 6 人版不同）

员工 1 与项目负责人确认以下整体交付规则。这些规则仍满足“一个 Issue 对应一个分支和一个 PR”，但把 15 个功能切片降为同一 Issue 内的任务清单或里程碑：

1. **一个总 Issue**：员工 1 的全部统一前端 P0 只创建一个总 Issue；15 个里程碑作为该 Issue 的可勾选任务清单，不再创建 15 个子 Issue。
2. **一个长期功能分支**：只创建 `feature/<issue>-unified-web-p0`，15 个里程碑全部在该分支持续实现，不再创建 15 个本地或远端功能分支。
3. **一个总 Draft PR**：第一个里程碑通过本地门禁后，将长期分支首次推送到 GitHub，并创建一个从该分支到 `main` 的 Draft PR；后续始终更新同一个 Draft PR。
4. **先本地测试，后推送**：每个里程碑的改动必须先通过该里程碑的本地测试和验收，才允许把对应提交推送到长期分支。未经本地门禁的代码不得上传 GitHub。
5. **前端整体交付**：所有前端页面最终由同一个长期分支、同一个总 PR 进入 `main`；里程碑推送只用于持续备份、CI 和评审，不代表分批合入。
6. **持续 CI 和评审**：每次里程碑推送后都运行同一个 Draft PR 的 CI，并允许评审人增量评审；失败必须在进入下一里程碑前处理。
7. **满足总门禁后才 Ready**：15 个里程碑全部完成、全量本地 E2E 通过、最终提交 CI 通过、测试环境验收通过后，才把总 PR 从 Draft 转为 Ready for Review。
8. **合并后统一关闭**：总 PR 获得至少一名非作者批准并合入 `main` 后，才关闭总 Issue 并删除长期功能分支。

后续章节（§6、§9、§13、§14、§15）均以这八条为准。“整体上传”指一个长期分支和一个总 PR，不指直到第 8 周才进行第一次远端备份。

```text

```

### 4.2 布局

```text
App
├── LoginLayout
└── AuthenticatedLayout
    ├── UserWorkspaceLayout
    └── AdminWorkspaceLayout
```

- 所有账号使用同一个登录入口。
- 管理员默认仍可进入普通用户工作区。
- 当前账号具有管理查看权限时，顶部显示"管理中心"。
- 管理中心使用 `/admin/*` 路由组。
- 普通用户手工访问 `/admin` 时显示 403，且不预加载管理数据。

### 4.3 权限

- 权限码由后端（员工 3）统一发放，前端不发明权限码。
- 不通过用户名、邮箱或固定角色名判断管理员。
- `GET /api/v1/me` 返回角色、权限码和知识库数据范围。
- 路由使用权限元数据。
- 按钮使用统一 `usePermission().can(permission)`。
- 前端隐藏只能减少误操作，后端必须最终鉴权。
- 权限撤销后退出失权页面，不保留旧管理数据。
- 路由与权限码映射以《知识库平台五人团队分工协作方案》§4.3 和员工 3 提交的 OpenAPI 权限目录为事实来源，不在本计划中另造权限码。
- 使用 HttpOnly Cookie 时，认证契约必须明确 `SameSite`、`Secure`、CSRF 防护和允许的 Origin；前端不得把 Cookie 认证误当作天然免疫 CSRF。

### 4.4 状态管理

Pinia 只保存：

- 当前用户和权限。
- 当前知识库。
- 当前会话的必要摘要。
- 真正跨页面的共享状态。

以下状态保留在页面或 feature：

- 表单。
- 弹窗。
- 页码和排序。
- 单页筛选。
- 上传临时列表。
- 单次导出配置。
- 页面加载和错误。

### 6.2 15 个里程碑清单

- 每个里程碑使用 `M01` 至 `M15` 的稳定编号和英文 slug。

- 里程碑完成不关闭总 Issue，也不把总 PR 转为 Ready。

### 6.3 唯一长期功能分支

- GitHub 创建总 Issue 后，用实际编号替换 `<issue>`，只创建一次 `feature/<issue>-unified-web-p0`。
- 该分支从当时最新的 `main` 创建，承载 M01–M15 全部工作。
- 不创建 `web-foundation`、`auth-rbac-shell` 等 15 个独立本地或远端分支；这些名称只作为里程碑 slug。
- 每个里程碑本地通过后，在同一长期分支形成范围清晰的提交并推送到同一个 Draft PR。
- 开始新里程碑前获取 `origin/main` 最新状态；若 `main` 已变化，在长期分支完成合并和冲突处理，并重跑受影响门禁。不得假定 squash 能消除冲突。
- 总 PR 合入 `main` 前，长期分支不得用于第二个 PR，也不得把部分里程碑单独合入。
- 总 PR 合并后才关闭总 Issue，并删除本地和远端长期功能分支。

## 9. 长期分支与每个里程碑的固定流程

### 9.1 里程碑开始前

- [ ] 总 Issue 中对应里程碑满足 §12 DoR。
- [ ] API 草案、权限码或“本里程碑无 API”结论明确。
- [ ] PRD、Tech-Spec、tokens、assets、固定 Mock 和验收截图明确。
- [ ] 验收、代表性失败输入和测试数据明确。
- [ ] 长期分支工作区没有不明改动。
- [ ] 已获取 `origin/main` 最新状态；存在影响本里程碑的变更时先在长期分支完成同步和回归。

### 9.2 只创建一次长期功能分支

治理 PR #6 合并并创建总 Issue 后，只执行一次：

```powershell
git switch main
git pull --ff-only origin main
git switch -c feature/<issue>-unified-web-p0
```

后续 M02–M15 始终留在该分支。开始新里程碑前执行：

```powershell
git switch feature/<issue>-unified-web-p0
git fetch origin
git status --short
```

若 `origin/main` 有新提交，先在长期分支合并 `origin/main`、解决冲突并重跑受影响检查。不得另建里程碑分支，也不得把未经本地门禁的提交推送到任何 GitHub 远端。

### 9.3 每个里程碑的本地门禁

首次安装、锁文件变化或干净环境验证时先运行：

```powershell
npm.cmd ci
```

每个里程碑提交和推送前运行：

```powershell
npm.cmd run typecheck:web
npm.cmd run lint:web
npm.cmd run test:web
npm.cmd run build:web
```

同时检查与本里程碑相关的：

- [ ] 1440px、1280px 和 375px。
- [ ] 加载、空、401、403、404 和网络失败。
- [ ] 长文本、长文件名、宽表格或长代码。
- [ ] 重复点击、快速切换和页面离开后的请求清理。
- [ ] 浏览器控制台无意外错误。
- [ ] Network 请求、字段和错误结构符合当前 OpenAPI。
- [ ] 总 Issue 已记录测试命令、结果摘要和必要截图；失败项已修复并重新验证。

### 9.4 同一个 Draft PR 的持续推送

M01 通过 §9.3 后：

1. 在长期分支提交 M01，提交信息使用项目约定格式。
2. 执行 `git push -u origin HEAD`。
3. 从 `feature/<issue>-unified-web-p0` 向 `main` 创建 Draft PR，标题为 `员工 1：统一前端 P0 完整交付`，并使用 `Closes #<issue>` 关联唯一总 Issue。
4. PR 描述建立 M01–M15 清单，记录本地测试、视觉证据、接口版本、风险和评审人。

M02–M15 每次通过本地门禁后：

1. 在同一长期分支提交该里程碑。
2. 执行 `git push origin HEAD`，更新同一个 Draft PR。
3. 等待该提交的 CI 结果，并邀请员工 3、4、5、6 增量评审各自主责部分。
4. CI 失败或出现阻断评审意见时，先在同一长期分支修复并重新通过本地门禁，再进入下一里程碑。

Draft 阶段的持续推送属于同一个整体交付，不得创建第二个前端 P0 PR，也不得提前合入部分页面。

### 9.5 从 Draft 转 Ready、合并和关闭

只有同时满足以下条件，才允许把总 PR 转为 Ready for Review：

- [ ] M01–M15 全部满足 §13。
- [ ] 长期分支已同步最新 `origin/main`，并在同步后重新通过 §10.4 全量本地 E2E。
- [ ] 员工 3、4、5 的最终 OpenAPI 契约已提交并与生成类型、Mock、测试一致。
- [ ] 最终提交对应的 CI `quality` 通过。
- [ ] 同一提交 SHA 的构建产物已部署到测试环境并通过 §10.5。
- [ ] 总 Issue 中没有未完成 P0、权限、安全或主流程阻断项。

转 Ready 后，由至少一名非作者批准；建议员工 3、4、5 分别确认自身接口，员工 6 确认 CI、部署和测试证据。总 PR 合并后才关闭总 Issue 并删除长期功能分支。若 `main` 要求线性历史，使用仓库允许的 squash 或 rebase 合并策略，不通过改写已共享历史规避评审。

## 10. 本地测试

### 10.1 Mock 模式

```powershell
npm.cmd ci
npm.cmd run dev:web
```

访问：

```text
普通用户：http://localhost:5173
管理中心：http://localhost:5173/admin
```

Mock 至少包含：

- 普通用户。
- 部分管理权限用户。
- 完整管理员。

### 10.2 真实接口模式

```powershell
docker compose -f deploy/docker-compose.yml up -d postgres redis api-server worker
npm.cmd run dev:web:api
```

API：

```text
http://127.0.0.1:8000/api
```

> 上述真实接口开发命令只启动 4 个后端/基础设施容器，Web 由本地 Vite 提供；完整 Compose 或测试环境才运行 5 个容器（`web`、`api-server`、`worker`、`postgres`、`redis`）。员工 6 须保证 Compose 不再包含已弃用的 `user-web` / `admin-web` / `shared-ui` 镜像，并提供健康检查验证方式。

### 10.3 权限测试矩阵

| 场景 | 预期 |
|---|---|
| 普通用户登录 | 不显示管理中心 |
| 普通用户访问 `/admin` | 403 且不请求管理数据 |
| 仅有一个管理模块查看权限 | 只显示该模块及只读操作 |
| 部分管理员 | 只显示获授权页面和按钮 |
| 完整管理员 | 用户功能和管理 P0 全部可用 |
| 权限被撤销 | 清理旧状态并退出失权页面 |
| 后端返回 403 | 不显示伪成功状态 |
| 登录过期 | 只执行一次刷新或安全退出 |
| 用户被禁用 | 会话按认证契约失效并清理业务状态 |

### 10.4 全量本地 E2E（总 PR 转 Ready 的硬性前置）

M15 完成后、总 PR 转 Ready 前，必须在长期功能分支的最终候选提交上跑通以下 10 条流程，并留下截图或录屏作为总 PR 证据：

1. 登录 → 加载个人信息 → 选择知识库 → 打开文档 → Markdown 预览 → 页码或引用定位。
2. 选择知识库 → 选择检索方式 → 输入问题 → 查看检索结果 → 打开来源文档。
3. 选择知识库 → 创建会话 → 输入问题 → 流式回答 → 停止或重新生成 → 引用 → 原文跳转。
4. 打开历史会话 → 搜索 → 重命名 → 继续提问 → 删除并确认路由和列表同步。
5. 选择当前文档和多文档 → 选择格式 → 创建导出任务 → 在“我的下载”查看进度 → 鉴权下载或查看过期状态。
6. 完整管理员登录 → 系统概览 → 用户、角色和权限管理，并验证自停用防护和失败提示。
7. 模型创建/编辑/启停/连通性测试 → 知识库创建/授权/参数修改，并确认密钥不回显。
8. 文档单个/批量上传 → 查看独立进度 → 原始文件/Markdown/分段预览 → 失败重试或人工处理。
9. 运行命中率测试 → 保留参数和请求编号 → 跳转来源；查询审计日志并确认敏感字段掩码。
10. 普通用户、单模块只读管理员、部分管理员、完整管理员依次验证路由、按钮、403、权限撤销和禁止预加载管理数据。

验收要求：

- 1440px 完整跑通 10 条流程；1280px 和 375px 对所有关键页面做响应式检查，并至少重跑普通用户主流程、管理主流程和权限主流程。
- 浏览器控制台无未处理错误。
- 流程终态或页面离开后无遗留业务请求、SSE 或轮询；Vite HMR 等开发连接不计入业务泄漏。
- Network 请求与最终 OpenAPI 一致，失败路径不出现伪成功或敏感信息泄露。

### 10.5 最终 CI 和测试环境验收

1. 将通过 §10.4 的最终候选提交推送到同一个 Draft PR。
2. 确认该提交对应的 CI `quality` 通过，不使用旧提交的绿色结果代替。
3. 员工 6 使用同一提交 SHA 构建并部署测试环境，不从开发者工作区手工复制页面文件。
4. 在测试环境重复 §10.4 的 10 条流程；允许减少重复截图，但不得减少权限、安全、下载和主流程覆盖。
5. 记录环境、提交 SHA、镜像标签、测试数据、结果、失败重试和剩余非阻断限制。
6. 测试环境通过后才允许把总 PR 转为 Ready。

## 11. 风险和处理

| 风险 | 处理 |
|---|---|
| 员工 1 单人承担全部前端 | P0 优先、15 个可验收里程碑、README、Mock 和测试；延误时先削减 P1 和非关键视觉增强 |
| 第 6、7 周里程碑过密 | 提前冻结接口和设计；DoR 未满足时记录阻塞并调整里程碑顺序，不压缩权限、安全和测试 |
| OpenAPI 延迟 | 草案未提交前只做静态 UI；草案提交后才生成类型和 Mock，真实契约未通过不得标记该里程碑联调完成 |
| 设计事实来源缺失 | PRD、Tech-Spec、tokens、assets、固定数据和验收截图未齐时不开始对应视觉实现 |
| 用户和管理组件互相污染 | 业务组件留在 feature，真正跨模块复用后再上移 |
| 权限显示和后端不一致 | 权限码单一来源（后端）、后端兜底、权限矩阵测试 |
| Cookie 认证遭受 CSRF | 后端明确 SameSite、Secure、CSRF 和 Origin 策略，前端按契约发送并测试失败场景 |
| Markdown XSS | 服务端清洗加 DOMPurify 二次过滤 |
| 模型密钥泄露 | 只写、不回显、不持久化、提交后清空 |
| SSE 断流或停止后继续计费 | 增量缓冲、半包和断流测试；OpenAPI/流式契约明确服务端取消语义 |
| 上传和导出轮询泄漏 | 终态、失败或页面离开时停止 |
| 长期分支与 `main` 漂移 | 每个里程碑前获取 `origin/main`，发生变化时在长期分支同步、解决冲突并重跑受影响门禁 |
| 未测试代码上传 | 每个里程碑先通过 §9.3，再提交和推送；CI 绿色不能替代本地门禁 |
| PR 评审不及时 | 第一次推送即创建 Draft PR，按里程碑持续邀请员工 3、4、5、6 增量评审，不把评审集中到第 8 周 |
| P1 插入首期 | 新需求默认进入第二阶段 Issue |
| Draft PR 变得过大 | PR 描述保持 M01–M15 清单，提交按里程碑分组，评审意见即时处理，禁止无关重构 |
| CI 环境差异 | 每个里程碑推送后持续运行 CI；最终提交仍需重新通过 `quality`，不复用旧结果 |
| M15 发现回归 | 直接在同一长期分支修复，重跑受影响测试和 §10.4 全量 10 条流程，不回到已废弃的里程碑分支 |

## 13. 里程碑完成定义

### 13.2 里程碑完成（推送后）

## 1. 项目目标

### 1.1 要解决的问题

平台面向需要集中管理内部文档并进行检索、问答和引用核验的团队，解决以下问题：

- 文档分散在 PDF、Office、图片、网页、表格和文本文件中，无法统一检索。
- 扫描件和图片没有可搜索文本，需要 OCR 和版面分析。
- 大模型回答缺少可核验引用，用户无法确认答案来自哪里。
- 用户、角色和知识库权限分散，容易发生越权访问。
- 文档转换、索引和导出耗时较长，不能依赖同步请求阻塞页面。
- 管理员缺少用户、模型、知识库、文档、任务和审计的统一管理入口。
- 多人开发容易出现接口字段、权限码、依赖版本和验收口径漂移。

### 1.2 首期产品目标

- 用户能够登录后查看获授权知识库和文档。
- 常见文档能够进入统一转换流程并生成标准 Markdown。
- 文档可以进行关键词、向量和混合检索。
- 用户能够获得流式回答，并查看文档、页码、原文片段和相关度。
- 管理员使用同一个 Web 应用进入管理中心，不维护第二套登录和基础组件。
- 管理员能够管理用户、角色、模型、知识库、文档、转换任务、命中率测试和审计日志。
- 文档可以导出为 PDF、DOCX、Markdown、CSV、TXT；多文档生成 ZIP。
- 项目能够通过 Docker Compose 部署，并具备健康检查、监控、备份和回滚说明。

### 1.3 成功指标

| 指标 | 首期目标 | 说明 |
|---|---:|---|
| P0 功能验收通过率 | 100% | P0 未通过不得以 P1 补偿 |
| 后端单元测试覆盖率 | 不低于 70% | 核心权限、任务状态和数据边界优先 |
| 支持格式自动化样本覆盖 | 每类至少 1 个正常样本和 1 个异常样本 | 样本不包含真实敏感数据 |
| 五种导出格式可读性 | 100% | CI 或测试环境完成基本打开和内容检查 |
| 管理接口权限校验 | 100% | 前端隐藏不能替代后端校验 |
| 生产部署镜像一致性 | 100% | 测试通过的镜像原样进入生产 |
| 严重安全问题 | 0 | 包括越权、Secret 泄露、任意文件访问和存储型 XSS |

### 2.1 普通用户 P0

## 3. 总体架构

### 3.1 逻辑架构

```text
                         统一 Web 应用
              普通用户工作区 + 管理中心 /admin
                              |
                            Nginx
                 静态资源、SPA fallback、/api 代理
                              |
                       FastAPI API Server
                              |
       +----------------------+----------------------+
       |                      |                      |
   认证权限模块           知识文档模块           RAG 与导出模块
       |                      |                      |
       +----------------------+----------------------+
                              |
                    PostgreSQL + pgvector
                              |
                       本地文件存储卷
                              |
                    Redis 异步任务队列
                              |
                            Worker
```

### 3.2 首期部署单元

| 容器 | 主要职责 | 暴露方式 | 健康检查 |
|---|---|---|---|
| `web` | 构建后的 Vue 静态资源、Nginx、SPA fallback、API 反向代理 | 对外 80/443 | 静态首页和 Nginx 状态 |
| `api-server` | FastAPI 同步接口和 SSE | 仅由 Nginx 访问 | `/api/v1/health/live`、`/api/v1/health/ready` |
| `worker` | 转换、OCR、分段、索引和导出 | 不对外 | 队列心跳和任务延迟 |
| `postgres` | 业务数据、全文检索、向量 | 仅内部网络 | `pg_isready` |
| `redis` | 队列、短期缓存、分布式锁 | 仅内部网络 | `redis-cli ping` |

### 3.3 架构约束

- 后端是一个工程、一个 API 服务，按 Python 包隔离业务模块。
- 模块之间通过服务层调用，不跨模块直接修改数据库表。
- 普通 CRUD、查询和检索请求使用同步 HTTP。
- 文档处理、OCR、索引、批量导出使用异步任务。
- 流式问答使用 `POST /api/v1/chat/stream` 和 SSE 语义返回。
- 原始文件和标准化 Markdown 都必须保存。
- API、Worker 和 Web 使用同一版本号和同一发布批次。
- 测试和生产使用相同镜像，仅环境变量和外部资源不同。

### 3.4 关键数据流

#### 文档数据流

```text
管理员上传
-> API 校验权限、文件大小、名称和 MIME
-> 保存原始文件并创建 document/task
-> Worker 识别解析器
-> 必要时 LibreOffice 转换
-> 必要时 OCR
-> 标准化 Markdown 和资源
-> 分段
-> 全文索引与向量索引
-> 文档状态 ready
-> 用户可检索、预览和问答
```

#### 问答数据流

```text
用户提问
-> 校验 chat.use 与知识库数据权限
-> 检索候选片段
-> 权限和元数据过滤
-> 重排
-> 生成带引用上下文
-> 模型流式生成
-> SSE 返回增量文本和引用
-> 保存会话、消息、引用和用量
```

#### 导出数据流

```text
用户选择范围和格式
-> 校验资源权限
-> 创建 export_task
-> Worker 生成文件
-> 保存导出结果和过期时间
-> 用户在我的下载中鉴权下载
-> 到期清理文件并保留必要审计记录
```

### 4.4 权限展示规则

- 前端不得通过用户名、邮箱或固定角色名称判断管理员。
- `GET /api/v1/me` 返回角色、权限码和知识库数据权限。
- 管理入口是否显示，由管理路由表中的查看权限集合决定，不使用字符串前缀猜测。
- 路由守卫检查页面权限；页面内按钮使用统一 `usePermission().can(permission)`。
- 前端隐藏按钮只减少误操作，后端仍对每次请求鉴权。
- 权限变更后，下一次 `/me` 刷新或接口 403 必须更新前端权限并退出失权页面。
- 多个请求同时出现 401 时只允许一个刷新请求，其余请求等待同一结果。
- 刷新失败时清空内存状态并跳转登录页，不在 URL 中携带 Token。

### 4.5 前端状态边界

Pinia 只保存：

- 当前用户和权限。
- 当前知识库。
- 当前会话摘要和必要的跨页状态。
- 全局通知需要的最小状态。

以下状态保留在页面或 feature：

- 表单内容和校验结果。
- 弹窗开关。
- 页码、排序和单页筛选。
- 上传列表和单个任务临时状态。
- 单次导出配置。
- 单个页面的加载、错误和空状态。

### 4.6 前端实现标准

- Vue 使用 Composition API 和 `<script setup lang="ts">`。
- TypeScript 开启 strict，禁止 `any`。
- 外部响应在 API 边界安全收窄，页面不直接处理 `unknown`。
- API DTO 由 OpenAPI 生成；前端内部视图模型可以显式转换，但不得复制后端契约。
- 普通 HTTP 请求复用统一 API Client。
- 流式问答使用原生 Fetch、ReadableStream、TextDecoder 和 AbortController，不增加专用 SSE 依赖。
- Markdown 使用 Markdown-It 渲染，并在展示前通过 DOMPurify 二次过滤。
- 视觉使用 Element Plus 和统一 tokens；不创建没有使用场景的设计系统抽象。
- 桌面端重点验收 1440px 和 1280px；375px 下核心功能可用、无关键遮挡。

## 5. 登录、角色与权限模型

### 5.1 核心实体

| 实体 | 关键字段 | 归属模块 |
|---|---|---|
| `users` | id、username、display_name、password_hash、status、last_login_at | auth/users |
| `roles` | id、name、description、status | users |
| `permissions` | id、code、name、module、action | users |
| `user_roles` | user_id、role_id | users |
| `role_permissions` | role_id、permission_id | users |
| `knowledge_base_permissions` | subject_type、subject_id、kb_id、access_level | knowledge/users |
| `sessions` 或 refresh token 记录 | user_id、token_hash、expires_at、revoked_at | auth |

### 5.2 登录流程

```text
提交用户名和密码
-> 后端验证状态和密码哈希
-> 写入登录审计
-> 设置安全 Cookie
-> 前端调用 /me
-> 加载角色、权限和知识库范围
-> 进入原目标页或用户首页
```

### 5.3 认证安全

- 密码使用 Argon2，不存储明文或可逆加密密码。
- Refresh Token 只保存哈希或等价安全表示。
- Cookie 使用 HttpOnly、Secure（生产）、适当 SameSite 和有限 Path。
- 登录、刷新、重置密码和连通性测试需要频率限制。
- 用户被禁用、密码重置或角色撤销后，已有会话按契约失效。
- 重置密码接口不得返回旧密码或在日志中记录新密码。
- 前端不读取、打印、复制或持久化 Token。

## 6. 文档上传与 Markdown 转换

### 6.1 “任意文档”的可交付定义

“任意文档”不是承诺任何未知格式都能自动解析，而是提供统一入口和可扩展处理机制：

- 同时检查扩展名、MIME 类型和文件头。
- 常见办公文档、文本、网页、表格、演示文稿、电子书、邮件和图片使用可插拔解析器。
- 图片和扫描 PDF 使用 OCR，并保留原图和置信度。
- 无法识别的文件保留原件，进入 `manual_review`，不静默丢弃。
- 新增格式时实现解析器适配器，不修改上传接口。

### 6.2 首期格式

| 类型 | 扩展名 | 处理方式 | 代表性失败 |
|---|---|---|---|
| PDF | `.pdf` | 原生文本；扫描件 OCR；提取页码、图片和表格 | 加密 PDF、损坏页、字体编码异常 |
| Word | `.docx`、`.doc` | DOCX 解析；DOC 经 LibreOffice 转换 | 宏、嵌入对象、转换超时 |
| Excel | `.xlsx`、`.xls` | 工作表转表格；大表生成附件引用 | 合并单元格、公式、超大工作表 |
| PowerPoint | `.pptx`、`.ppt` | 每页标题、正文、备注和图片 | 旧格式转换失败、复杂动画 |
| Markdown | `.md`、`.markdown` | 清洗标题、链接、代码和图片 | 危险 HTML、外部跟踪链接 |
| 文本 | `.txt`、`.log`、`.rst`、`.org` | 编码识别并生成段落或代码块 | 编码无法识别、超长行 |
| 表格文本 | `.csv`、`.tsv` | Markdown 表格并保留原件 | 编码、分隔符、列数不一致 |
| 网页与结构化文本 | `.html`、`.htm`、`.xml`、`.json` | 提取正文和层级结构 | 脚本、巨大嵌套、非法结构 |
| 电子书与开放文档 | `.epub`、`.odt`、`.ods`、`.odp`、`.rtf` | 中间格式转换 | 资源缺失、章节损坏 |
| 图片 | `.jpg`、`.jpeg`、`.png`、`.webp`、`.bmp`、`.tif`、`.tiff` | OCR、版面分析和原图引用 | 低清晰度、旋转、语言不匹配 |
| 邮件 | `.eml` | 主题、发件人、正文和附件清单 | 损坏 MIME、超大附件 |

### 6.3 统一流程

```text
uploaded
-> detecting
-> converting
-> ocr（必要时）
-> normalizing
-> chunking
-> indexing
-> ready
```

异常终态：

```text
failed
manual_review
cancelled
```

### 6.4 幂等和重试

- 每个原始文件计算哈希，用于去重和版本识别。
- `duplicate_policy` 支持 `skip`、`new_version`、`replace`。
- 每个任务使用稳定的业务幂等键，重复执行不得产生多份有效索引。
- 重试从可安全恢复的阶段开始；无法判断时重新构建该文档的派生数据。
- 新索引完全成功前保留旧有效索引，避免用户看到半成品。
- 任务记录保存阶段、进度、重试次数、安全错误、请求编号和开始/结束时间。

### 6.5 Markdown 包规范

```text
storage/documents/{document_id}/
├── original/
├── markdown/
│   └── content.md
├── assets/
├── preview/
└── manifest.json
```

`content.md` 要求：

- 文档标题使用一级标题，正文从二级标题开始。
- 普通表格使用 Markdown；复杂表格允许经过清洗的 HTML。
- 图片使用相对路径，例如 `![说明](../assets/image-001.png)`。
- OCR 内容保留页码和置信度元数据。
- 分页位置提供稳定锚点，供引用定位。
- 删除脚本、事件属性、危险 URL 和跟踪代码。
- 保留文件哈希、解析器名称、解析器版本和转换时间。
- `manifest.json` 记录页码、标题、资源、原始文件、哈希和解析警告。

### 6.6 分段规范

- 优先按 Markdown 标题、段落和列表边界切分。
- 表格和代码块尽量保持完整。
- 每个 Chunk 保存文档、章节、页码、顺序号和字符范围。
- Chunk 参数由知识库配置提供，服务端验证最小值、最大值和 overlap 合法性。
- 索引内容不包含用户无权访问的隐藏资源。

## 7. 检索、问答与引用

### 7.1 检索方式

| 模式 | 主要用途 | 实现 |
|---|---|---|
| 关键词 | 精确名词、编号、原文匹配 | PostgreSQL 全文检索 |
| 向量 | 语义近似表达 | pgvector 相似度 |
| 混合 | 同时兼顾精确和语义 | 两路召回、归一化、融合和可选重排 |

### 7.2 检索处理顺序

```text
请求校验
-> 用户和知识库权限过滤
-> 元数据过滤
-> 关键词/向量召回
-> 分数归一化和融合
-> 可选重排
-> TopK 和阈值过滤
-> 返回来源、页码、片段和分数
```

权限过滤必须在生成最终结果前完成，不能先将无权内容发送给模型后再从前端隐藏。

### 7.3 流式问答事件

`POST /api/v1/chat/stream` 使用以下事件语义：

| 事件 | 关键字段 | 说明 |
|---|---|---|
| `start` | conversation_id、message_id、request_id | 建立本次回答上下文 |
| `delta` | text | 增量回答文本 |
| `citation` | citation_id、document_id、document_name、page_no、quote、score | 可核验引用 |
| `done` | finish_reason、usage | 正常结束 |
| `error` | code、message、request_id、retryable | 安全错误 |

前端使用增量缓冲解析，必须覆盖半包、多事件同包、UTF-8 字符被拆分、服务端断流和用户取消。

### 7.4 回答状态

```text
idle -> submitting -> streaming -> completed
                         |
                         +-> stopped
                         +-> failed
```

- `stopped` 保留已生成内容并明确标记未完成。
- `failed` 保留可安全展示的部分内容和重试入口。
- 页面离开时终止客户端请求；服务端检测断开并停止不必要生成。
- 重新生成创建新的回答版本，不覆盖原回答审计信息。

### 7.5 引用要求

- 每条引用必须关联当前用户有权访问的文档。
- 引用保存消息、Chunk、文档、页码、原文片段、顺序和分数。
- 前端显示的片段来自服务端引用记录，不从回答文本猜测。
- 文档删除、归档或失权后，历史引用显示不可访问状态，不泄露原文。
- 回答没有足够证据时应明确说明，不伪造引用。

## 8. 导出与下载

### 8.1 首期格式

| 格式 | 内容 | 基本验证 |
|---|---|---|
| PDF | 目录、标题、页眉页脚、分页、图片和引用 | 可打开、中文字体正常、页数大于 0 |
| DOCX | 可编辑标题、段落、表格和图片 | 可打开、核心文本存在 |
| Markdown | 标准 Markdown 和资源压缩包 | UTF-8、链接和资源路径正确 |
| CSV | 元数据、章节、页码、Chunk 和正文 | UTF-8 BOM 策略固定、列结构一致 |
| TXT | 去除 Markdown 标记后的纯文本 | 编码和换行正常 |

### 8.2 首期导出范围

- 当前文档。
- 多选文档。

第二阶段扩展：

- 单条回答。
- 完整会话。
- 当前检索结果。

### 8.3 CSV 字段

```text
document_id,document_name,section_no,heading,page_no,chunk_no,content,source_url,tags
```

字段顺序固定；不同页面不得实现不同 CSV 结构。

### 8.4 任务和下载安全

- 小型任务也通过统一导出服务，页面不自行拼装最终文件。
- 多文档导出生成 ZIP。
- 下载接口实时鉴权或签发短期有效地址。
- 导出任务保存创建人、范围、格式、状态、进度、结果、过期时间和安全错误。
- 前端在终态、页面离开或连续失败时停止轮询。
- 文件到期后删除物理文件，保留必要的任务和审计元数据。

## 15. 安全要求

### 15.1 权限与越权

- 每个资源查询都应用用户、角色和知识库数据权限。
- 不允许通过修改 UUID 查看或下载无权资源。
- 管理路由和按钮显示不能替代后端权限。
- 批量接口逐项或整体校验权限，失败策略写入 OpenAPI。
- 审计记录权限变更、模型配置、文档删除和导出。

### 15.2 Secret

- 不读取、打印、复制或提交 `.env`、Token、API Key、密码或私钥值。
- 模型密钥通过安全配置或加密存储管理。
- API 不回显模型密钥。
- 前端提交密钥成功后立即清空输入状态。
- 日志和错误不得包含 Authorization、Cookie、密码或密钥字段。

### 15.3 文件安全

- 清理文件名并阻止路径穿越。
- 扩展名、MIME 和文件头交叉校验。
- 上传目录禁止执行。
- 解析工具使用资源和超时限制。
- 原始文件和导出文件不通过公开静态目录暴露。
- 压缩包解压限制文件数量、总大小和嵌套深度。

### 15.4 内容安全

- Markdown 服务端清洗、前端 DOMPurify 二次过滤。
- 禁止脚本、事件属性、危险协议和外部跟踪代码。
- 外部链接增加安全属性。
- 不使用 `eval()` 或任意代码执行。
- 模型输出视为不可信内容，不直接注入 HTML。

### 15.5 日志和错误

- 所有请求具有 `request_id`。
- 日志记录动作、主体、资源、结果和耗时，不记录敏感正文。
- 用户看到安全错误信息；详细堆栈只在受控服务日志中。
- 公开错误不得暴露存储路径、SQL、模型密钥或内部地址。

## 17. CI/CD、监控与发布

### 17.1 CI 必需 Job

工作流名称 `CI`，必需 Job 名称 `quality`。

前端出现后运行：

```text
npm ci
npm run typecheck:web
npm run lint:web
npm run test:web
npm run build:web
```

后端出现后运行：

```text
uv sync --project backend --frozen
ruff
mypy
pytest
```

### 17.2 镜像和发布

- Web、API、Worker 使用不可变版本标签和提交 SHA。
- 禁止生产使用 `latest`。
- 测试环境验收的同一镜像进入生产。
- 发布前检查数据库迁移、备份、健康检查和回滚镜像。
- 数据库迁移先评估前后兼容性；破坏性迁移分阶段执行。
- 发布失败时优先回滚应用镜像，数据库按预先验证方案处理。

### 17.3 Nginx 要求

- `/` 和 `/admin/*` 使用同一 SPA 产物。
- 未命中静态文件时 fallback 到 `/index.html`。
- `/api` 反向代理到 API。
- SSE 关闭代理缓冲并设置合适超时。
- 上传大小限制与后端配置一致。
- 生产使用 TLS 和安全响应头。

### 17.4 监控

至少监控：

- API 请求量、错误率和 P95 延迟。
- SSE 连接数、失败和取消。
- 文档任务队列长度、等待时间、失败率和重试。
- OCR、转换和索引耗时。
- 检索延迟和无命中率。
- 模型请求成功率、耗时和 Token。
- 导出任务耗时和失败率。
- PostgreSQL 连接、磁盘和慢查询。
- Redis 内存、队列和连接。
- 文件存储容量。

### 17.5 备份和恢复

- PostgreSQL：定期逻辑或物理备份，按环境定义保留期。
- 原始文件、Markdown、assets 和 manifest：同一恢复点策略。
- 向量索引可重建，但恢复时间需要评估。
- 备份不记录 Secret 明文。
- 第 8 周在测试环境执行恢复演练，并记录时间、步骤和结果。

## 19. 首期验收标准

### 19.1 普通用户

- 登录、刷新和退出正确。
- 只能看到有权知识库和文档。
- 三种检索可用，结果包含来源、页码、片段和分数。
- 流式问答可停止、重试和重新生成。
- 引用可定位原文；失权后不泄露内容。
- Markdown、表格、代码和图片显示正常且安全。
- 会话可搜索、重命名、删除和恢复。
- 当前文档和多文档可导出五种格式。
- 下载文件名、中文编码、内容范围和权限正确。

### 19.2 管理员

- 管理员同时可用用户功能和管理中心。
- 普通用户不显示管理中心，访问管理 URL 返回 403。
- 部分管理员只显示有权模块和按钮。
- 用户、角色、模型、知识库和文档具备 P0 管理能力。
- 文件上传逐项显示状态和失败原因。
- 管理员可预览原始文件、Markdown 和资源。
- 失败任务可重试，人工处理状态清晰。
- 命中率测试参数和结果可复现。
- 管理操作能在审计日志中查询。

### 19.3 文档处理

- 格式表中的文件进入统一任务流程。
- PDF、Word、Markdown、CSV、TXT 和常见图片可生成 Markdown。
- 图片和扫描 PDF 可生成可搜索文本。
- Markdown 保留标题、段落、表格、图片和来源位置。
- 原始文件、Markdown、assets 和 manifest 可追溯。
- 失败显示阶段、安全错误和重试入口。
- 重试不会产生重复有效索引。

### 19.4 工程质量

- 所有代码通过 PR 合并。
- 后端覆盖率不低于 70%。
- 核心接口具备自动化测试。
- OpenAPI、前端类型和 Mock 一致。
- 文档格式和五种导出具备回归样本。
- 单前端生产构建通过。
- 测试和生产使用同一镜像。
- 数据库、原始文件和 Markdown 具备备份恢复方案。
- 1440px、1280px 和 375px 完成视觉验证。

## 21. 开始条件与完成定义

### 21.1 Definition of Ready

一个 Issue 开始前必须满足：

- 用户价值和目标明确。
- 包含和不包含范围明确。
- 负责人和协作人明确。
- API 草案、权限码或无 API 结论明确。
- 页面或任务状态明确。
- 验收和测试样本明确。
- 依赖分支已合并或不存在依赖。
- 没有要求读取、打印或提交 Secret。

### 21.2 Definition of Done

一个功能只有同时满足以下条件才能关闭：

- 实现与 Issue 范围一致，没有夹带无关重构。
- 代码、配置和文档进入对应分支。
- OpenAPI、生成类型、Mock 和测试同步。
- 权限、输入、异常、空状态和日志已处理。
- 类型检查、Lint、测试和构建通过。
- UI 完成桌面和移动端代表性检查。
- 数据库变更包含迁移和回滚。
- PR 关联 Issue，至少一名非作者批准。
- CI 通过，测试环境验收通过。
- 必要的部署、监控和回滚说明已更新。
- 长期架构决定或踩坑已更新 `MEMORY.md`；没有更新时说明原因。

## 22. 第二阶段规划

P1 在首期发布后重新评估，不直接沿用未确认的时间估算。优先顺序建议：

1. 用户反馈和检索质量闭环。
2. 回答、会话和检索结果导出。
3. 收藏。
4. 评测中心和批量模型对比。
5. 系统设置和模板管理。
6. Python 版本升级。
7. 只有容量和组织边界证明需要时再评估微服务拆分。

## 23. 最终责任原则

1. 每个模块只有一名主负责人，备份人不与主负责人并行修改同一范围。
2. 员工 1 负责全部前端体验和集成；后端负责人对权限、数据和业务正确性负责。
3. 管理员使用同一 Web 应用，管理功能只按权限显示，后端始终兜底鉴权。
4. 文档处理模块输出标准 Markdown、资源、manifest 和 Chunk；RAG 只通过公开服务层使用结果。
5. 导出统一由服务端导出模块处理，禁止各页面实现不同格式转换。
6. OpenAPI 是接口事实来源，禁止靠聊天消息或临时字段维持联调。
7. 所有直接依赖精确锁定，不使用浮动版本或 `latest`。
8. 所有代码、接口、测试、部署和最终文档进入 Git，不通过聊天工具传递最终版本。
9. 每次上线必须能回滚应用镜像，并具备数据库和文件恢复方案。
10. 安全、权限、可恢复性和 P0 测试不能为了赶进度被降级。

# 全项目一致性与可靠性审计（2026-07-20）

## 目标

逐层核对统一 Vue 前端、FastAPI 模块化单体、Worker、PostgreSQL/pgvector、Redis、Docker、CI 和项目文档，修复可以用本地证据确认的错误、契约漂移与可靠性问题。

## 范围与非目标

- 范围：`frontend/web/`、`backend/`、`deploy/`、`scripts/`、`.github/`、根配置及 `docs/`。
- 允许：精准修复已复现的问题，补充最小回归测试，同步受影响的事实来源和长期项目记忆。
- 非目标：新增产品功能、重做既有视觉基线、升级无关依赖、修改生产或 GitHub 状态。
- 安全边界：不读取或输出任何 `.env`、Token、API Key、密码或私钥值；外部模型能力只验证无凭据也可完成的边界行为。

## 验收标准

1. 前端类型检查、Lint、单元/组件测试和生产构建通过。
2. 后端 Ruff、Mypy、测试和 Alembic 迁移状态通过；新增或修复的非平凡逻辑有回归测试。
3. Docker 五服务健康，前端、API 健康接口和 OpenAPI 文档可访问，PostgreSQL、Redis 与 Worker 运行态可验证。
4. OpenAPI、前端服务调用、后端路由、权限和错误语义一致；如需改接口，先改契约再改实现与测试。
5. README、运维文档、固定版本、启动命令和当前真实实现一致，仓库内链接有效。
6. `git diff --check` 通过，改动不包含 secret、生成缓存或与审计无关的重构。

## 风险与约束

- Docker 运行态可能包含持久数据；默认只做读取、健康检查和测试，任何迁移或重建前先确认静态必要性。
- 外部 DeepSeek、DashScope 行为依赖本地 secret 与网络，不以调用外部付费 API 作为本次完成条件。
- 项目已有 501 个跟踪文件；“全量”指覆盖所有主要边界和自动化门禁，不把纯风格偏好当作缺陷。
- 工作区初始状态为 `main` 与 `origin/main` 一致且无本地改动；本次只保留可追溯到审计目标的本地修改，不提交、不推送。

## 修改原则

- 优先复用现有 helper、测试夹具和平台能力，不新增依赖或一次性抽象。
- 修根因；同类调用汇聚到共享边界时只修共享边界，不在各调用方重复打补丁。
- 仅修复有源码、测试、日志或运行态证据支持的问题；无法本地证明的建议记录为剩余风险，不臆测实现。

## 审计结论

- 未发现 P0。独立终审确认的 4 个 P1 和 8 个 P2 均已按最小可靠路径处理，并新增对应回归测试或运行态验证。
- 修复集中在认证与权限、数据约束、文件与任务生命周期、外部请求边界、API 契约、容器运行可靠性和前端真实联调；未新增产品功能，也未重做既有视觉基线。
- 所有改动仅保留在本地工作区，未提交、未推送，也未修改云端或 GitHub 状态。

## 剩余限制与风险

- 未调用真实 DeepSeek 或 DashScope 付费接口；只验证配置、权限、网络边界、无密钥回退和本地 RAG 链路。正式发布前仍需在受控环境执行一次真实模型冒烟。
- 样本清单仍有 20 个扫描件、旧 Office、EPUB、图片/OCR 等二进制样本待制作；它们明确标记为待制作，不计作通过。
- 边缘限流当前按来源 IP 和端点执行；同一 NAT 后的多个用户共享额度。若公开部署面对高并发或计费滥用，应再增加 Redis 支持的按用户配额。
- 上传补偿覆盖应用可见异常；数据库提交与文件移动无法形成真正的跨资源原子事务，主机在极窄提交窗口硬崩溃仍需依赖后续孤儿扫描或对象存储事务化方案。
- OpenAPI 已生成并被前端采用，但部分历史端点仍把成功数据声明为自由 `dict/object`，相应前端适配类型无法完全由契约生成；后续修改这些端点时应先补显式 Pydantic 响应模型。
- 未在真实业务数据上执行破坏性的恢复演练；本次只验证备份/恢复脚本的语法、参数、安全门禁和测试环境行为。

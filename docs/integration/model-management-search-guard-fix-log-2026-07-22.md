# 模型管理与 AI 搜索输入拦截修正记录

## 修正结论

本次修正解决了模型配置保存后返回 500、模型测试不能证明所选模型可用、RAG 未完整采用模型参数，以及敏感输入只能在结果页显示错误的问题。明显的涉黄赌毒输入现在会在搜索框内即时提示并禁用检索按钮；FastAPI 输入守卫继续作为不可绕过的最终边界。

## 问题与根因

| 问题 | 根因 | 用户影响 |
| --- | --- | --- |
| 模型配置保存报错 | 两次事务提交后直接序列化数据库更新时间，异步 ORM 隐式加载触发 `MissingGreenlet` | 配置实际可能已落库，但页面收到 500 |
| 模型测试误报可用 | 原测试只访问供应商 `/models`，没有调用所选聊天、Embedding 或 Rerank 模型 | 供应商可访问不等于模型可用于 RAG |
| RAG 模型配置未完整生效 | 回答固定使用全局温度和输出长度，Provider 方法未采用调用方超时 | 管理页面参数与实际推理不一致 |
| 敏感输入跳转后才报错 | 只有后端输入守卫，首页提交前没有确定性校验 | 用户进入结果页后才看到故障状态 |
| 后端测试构建过慢 | `backend/.venv` 和工具缓存进入 Docker 上下文，并递归修改大型依赖目录所有权 | 构建时间远大于测试执行时间 |

## 修改记录

### 后端

- 模型与供应商写接口在最终提交后执行 `AsyncSession.refresh()`，再构造响应。
- 模型测试按类型执行真实最小调用：聊天补全、单条向量生成或单文档重排。
- Embedding 测试校验实际维度；空聊天回答和空重排结果均判定失败。
- Provider 请求采用调用方传入的超时，默认模型请求超时调整为 30 秒并可通过环境变量覆盖。
- RAG 回答读取所选聊天模型的 `temperature` 和 `max_tokens`，同时保留安全范围限制。

### 前端

- 新增统一查询安全函数，对确定性涉黄赌毒词执行 NFKC 归一化和标点规避检测。
- 首页与结果页共用的搜索框显示“输入内容不合法，请修改后重试”，设置 `aria-invalid` 并禁用检索按钮。
- 结果页执行函数再次校验，覆盖路由状态和其他组件直接调用；后端输入类 422 只显示通用警告，不渲染红色故障面板。
- 模型测试文案改为“测试模型真实调用”，避免与旧的供应商连通性检查混淆。

### 构建

- Docker 递归排除子目录 `.venv`、Pytest、Mypy、Ruff 和 Python 缓存。
- API/Worker 只开放存储目录写权限，不再递归 `chown` 全部代码和大型依赖。

## 验证结果

- 前端：95 项 Vitest 全部通过，ESLint 零警告，TypeScript 检查和 Vite 生产构建通过。
- 后端：387 项中 385 通过、2 项按环境条件跳过；Ruff、Mypy、迁移检查和样本校验通过，覆盖率 62.11%。
- Docker：构建上下文由约 1.84 GB 降至 33 KB；同一完整后端门禁由约 702 秒降至 91 秒。
- 运行态：API、Worker、Web、PostgreSQL、Redis 全部健康，就绪检查返回 200。
- 运行态：DeepSeek 真实模型测试返回成功；原配置保存接口返回 200；真实 RAG 使用 `deepseek-v4-pro` 生成带来源答案。
- 浏览器：敏感输入不跳转且按钮禁用；合法“医院病毒防控”问题通过校验并只需提交一次。

## 剩余限制

- 前后端只保留确定性关键词规则；后端模型守卫因正常医疗问题误判已撤除，不再覆盖复杂
  改写主题和提示词注入。
- “测试模型真实调用”和 RAG 验收都会产生一次外部模型调用及相应计费。
- 生产环境仍应独立配置 `MODEL_KEY_FERNET_KEY`，避免轮换 `SECRET_KEY` 后无法解密已保存的模型密钥。

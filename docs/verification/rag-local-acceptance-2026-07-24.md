# RAG 本地部署验收记录（2026-07-24）

## 验收范围

- Docker Compose 全栈部署：Web、FastAPI、Worker、PostgreSQL + pgvector、Redis。
- 登录、部门知识库可见性、文档上传处理、Markdown 切分、向量写入、重处理。
- 混合检索、重排、流式生成、引用、缓存和无依据问题拒答。
- 管理看板与用户激励接口、OpenAPI 文档和前后端生产构建。

## 运行环境

- 用户入口：`http://127.0.0.1/`
- FastAPI 文档：`http://127.0.0.1/api/v1/docs`
- 检索数据：医疗信息化知识库，15 份就绪文档、587 个活动分块。
- 向量状态：587/587 个活动分块均写入 1536 维向量。
- 模型连通：聊天、Embedding、Rerank 三类真实模型均通过连通性测试。

## RAG 准确性

使用 5 条医疗信息化人工复核问题进行 Hybrid + Rerank 测试。测试集原有相关分块标签已过期，
先按当前活动分块人工复核并更新标签，再执行正式验收。

| 指标 | 结果 |
| --- | ---: |
| Hit Rate | 1.0000 |
| MRR | 0.9000 |
| NDCG@K | 0.9262 |
| Recall | 1.0000 |
| MAP | 0.9000 |
| Precision@5 | 0.2000 |

`Precision@5` 较低是因为每个问题仅标注 1 个相关分块，而接口固定返回 5 个候选，并不代表
Top 1 检索错误。5 条问题的 Top 1 主题人工复核均正确。

## RAG 速度

| 模式 | 平均/典型耗时 | 准确性观察 |
| --- | ---: | --- |
| Keyword，无重排 | 约 40 ms | Top 1 主题命中约 40% |
| Hybrid，无重排 | 约 1.95 s | Top 1 主题命中约 60% |
| Hybrid + Rerank | 约 3.0–3.7 s | Top 1 主题命中 100% |
| 流式回答，冷请求 | 首个事件约 14 ms；首段答案约 3.8 s；总计约 5.6–12.1 s | 回答有引用 |
| 完全相同问题缓存 | 缓存检查约 11–28 ms；接口总计约 35 ms | 复用已完成答案 |

交付配置保留 Hybrid + Rerank，以约 1–2 秒的额外开销换取明显更高的 Top 1 准确性。
缓存能覆盖重复演示问题，且 Redis 失效时会降级为正常 RAG。

## 真实链路验证

1. 上传 Markdown 测试文档并进入 Worker 队列。
2. 文档完成清洗、Markdown 标准化、切分和 Embedding。
3. 生成 2 个活动分块，2/2 均写入真实向量。
4. Hybrid + Rerank 能检索到新文档，Top 1 为预期章节。
5. 重处理任务约 6.2 秒完成，状态为 `ready`、进度 100%。
6. 非医疗问题不会编造依据，回答明确提示知识库中缺少相关内容。

## 本次修正

- 检索投影优先读取真实 `embedding_vector`，不再把 8 维旧兼容向量写入 1536 维 pgvector。
- 文档处理流水线增加事务保存点，单个步骤失败后仍能可靠记录失败状态。
- 重处理幂等键缩短到数据库字段上限以内，避免任务创建失败。
- 管理看板和用户激励分页参数由 `Literal[int]` 改为 `IntEnum`，修复合法查询串返回 422。
- 校验异常日志只记录字段位置和错误类型，不记录用户原始输入。
- 用户区“管理中心”入口按权限跳到首个可访问页面，知识库编辑者不再先进入无权看板并收到 403。
- 本地持久化演示账号名由历史值 `qmxh` 对齐为种子定义 `qmxl`。

## 剩余限制

- 首次真实模型请求受供应商网络影响，冷请求延迟会波动。
- 测试集只有 5 条医疗问题，适合交付演示，不等同于生产规模评测；上线前建议扩展到
  50–100 条并覆盖同义改写、跨文档问题、无答案问题和权限隔离问题。
- 云环境必须单独配置 `MODEL_KEY_FERNET_KEY`，不能继续依赖开发环境回退值。

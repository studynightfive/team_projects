# 业务看板与用户激励机制 PRD

## 1. 背景

当前管理首页主要展示用户数、知识库数和文档数，检索成功率、响应时间等字段仍为占位值，
无法回答“知识是否可用、回答是否有效、系统是否提速”等业务问题。普通用户也无法看到自己
对知识库建设的真实贡献。

本需求将管理首页改为基于真实检索和文档数据的业务看板，并提供不鼓励刷搜索次数的贡献积分、
徽章和部门榜单。

## 2. 目标

- 管理员可按最近 7、30、90 天和部门查看业务 KPI。
- KPI 包含知识覆盖、活跃检索、有效回答、未命中、文档处理成功率、缓存命中率和响应时间。
- 普通用户只能查看自己的积分、徽章、部门排名和贡献明细。
- 超级管理员可查看全部部门；部门管理员只能查看本部门。
- 列表支持分页，并具有加载、空数据和失败状态。
- 所有指标来自真实 API 和数据库，不使用固定样例。

## 3. 指标口径

| 指标 | 口径 |
| --- | --- |
| 知识覆盖率 | 统计范围内，至少有一份处于 `ready` 且启用索引文档的有效知识库数 / 有效知识库总数 |
| 活跃检索 | 统计期内完成的搜索和回答请求数 |
| 有效回答 | 统计期内命中至少一个分块且完成生成，或命中可复用答案缓存的回答数 |
| 未命中 | 统计期内未命中任何分块且未生成回答的回答数 |
| 文档处理成功率 | 统计期内进入 `ready` 的文档数 / 进入 `ready` 或 `failed` 的文档数 |
| 缓存命中率 | 统计期内命中答案缓存的回答数 / 已完成回答数 |
| 平均响应时间 | 统计期内完成检索和回答的平均耗时 |

指标从本功能上线后新增的结构化检索记录开始累计。历史审计日志缺少命中数和缓存状态，
不反向猜测历史指标。

## 4. 积分与徽章

### 4.1 积分规则

- 一份成功处理并启用索引的文档计 10 分。
- 同一用户上传相同内容时，以 `created_by + content_hash` 去重，只计一次。
- 个人知识库和企业知识库中的有效文档均可计分。
- 搜索、刷新、重复处理和失败文档不计分，避免通过高频操作刷分。

贡献明细直接关联文档标题、文档 ID 和生效时间，管理员可通过文档与审计记录复核。

### 4.2 徽章规则

| 徽章 | 条件 |
| --- | --- |
| 知识新星 | 累计积分达到 10 |
| 知识共建者 | 累计积分达到 50 |
| 知识领航员 | 累计积分达到 100 |

徽章按当前真实积分动态计算，不单独维护可漂移状态。

### 4.3 部门榜单

- 按部门内用户积分总和排序。
- 同分时按有效贡献文档数降序、部门名称升序。
- 普通用户只看到自己在所属部门内的个人排名，不返回其他用户明细。

## 5. 权限

- `admin.dashboard.view`：可访问管理看板。
- 超级管理员：可查看全部部门或选择单个部门。
- 部门管理员：只能查看本部门；请求其他部门返回 403。
- 已登录用户：只能访问 `/api/v1/me/incentives` 获取自己的激励数据。

## 6. 验收标准

1. 完成一次真实搜索或回答后，看板检索指标随统计范围变化。
2. 缓存回答、未命中回答和正常生成回答分别进入正确口径。
3. 文档从处理中进入 `ready` 后计入处理成功率和贡献积分。
4. 重复上传相同内容不会重复计分。
5. 超级管理员可切换部门，部门管理员不能越权。
6. 普通用户只能看到自己的积分、徽章、排名和分页贡献明细。
7. 前端在 1440px、1280px 和 375px 下无重叠，并覆盖加载、空和失败状态。

## 7. 非目标与限制

- 本期不实现积分兑换、人工加减分和消息推送。
- 本期不为搜索次数计分。
- 历史检索审计数据不具备完整业务字段，因此不补算历史命中率和缓存命中率。

# 撤除后端 LLM Guard 模型扫描 PRD

## 背景

FastAPI 当前在每次检索和问答前依次运行 `BanTopics` 与 `PromptInjection`。医疗信息化
问题容易被零样本主题分类或提示词注入模型误判，导致合法问题返回 422；CPU 推理也明显
增加首次启动和每次查询的等待时间。

## 目标

- 撤除 FastAPI 中由 `llm-guard` 执行的模型扫描、预热和线程池。
- 合法医疗信息化问题直接进入检索，不再经过语义主题或提示词注入模型。
- 保留确定性的涉黄、涉赌、涉毒关键词校验，防止绕过前端直接调用 API。
- 移除不再使用的 PyTorch、模型缓存、环境变量与就绪检查项。
- 不改变检索、RAG 回答、流式问答和会话接口契约。

## 验收标准

1. 正常医疗问题及带有一般指令表达的问题通过 `ensure_safe_query`。
2. 明确命中涉黄、涉赌、涉毒词表的输入仍返回统一 422，且不访问 RAG 下游。
3. 后端不再导入 `llm_guard`、加载模型、创建守卫线程池或等待模型预热。
4. `/api/v1/health/ready` 只检查数据库、Redis 与文件存储。
5. 后端依赖和锁文件不再包含 `llm-guard`、PyTorch 及其仅由守卫引入的依赖。
6. Docker Compose 不再创建或挂载 LLM Guard 模型缓存卷。
7. 后端 Ruff、mypy、测试及 Docker 运行态健康检查通过。

## 边界与风险

- 撤除后不再提供基于模型的提示词注入识别和敏感主题语义泛化。
- 确定性规则只能处理明确词语；生产环境仍需结合账号权限、审计、模型供应商内容安全
  和网关/WAF 策略进行纵深防护。
- 前端校验只改善交互，后端确定性校验仍是直接 API 调用的最终边界。

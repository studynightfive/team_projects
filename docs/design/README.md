# 设计事实来源

本目录保存进入前端实现前必须确认的产品与视觉输入。它解决的是“实现依据是否唯一”，不是用截图替代可编辑设计，也不承载生产代码。

## 事实来源顺序

同一里程碑内发生冲突时，按以下顺序处理：

1. 已批准的 `PRD.md`：用户价值、范围、信息架构和可观察验收。
2. 已批准的 `Tech-Spec.md`：实现边界、响应式、可访问性和状态归属。
3. `tokens.css`、`assets/manifest.md`、`mock-data.json`：可复用视觉值、真实资产策略和确定性设计数据。
4. `artifact.html`：可编辑、可离线查看的静态视觉源。
5. `acceptance/`：由同一提交的静态视觉源导出的验收证据。

PNG 只用于比对，不得反向覆盖 PRD、Tech-Spec、tokens 或静态视觉源。需要改变产品范围、文案、视觉值或资产策略时，必须先修改上游事实来源，再重新生成验收图。

## 审批边界

- 项目负责人通过非作者 PR 评审批准产品文案、视觉值、信息层级和资产策略。
- 员工 1 负责技术可实现性、前端结构、响应式与截图 QA。
- 出现真实 API、认证字段或权限码时，必须由员工 3 的 OpenAPI 或权限目录确认；设计 Mock 不具有契约效力。
- 员工 6 评审固定运行时、CI 命令、构建边界和验收证据的可重复性。

未完成上述批准的视觉值只能称为提案，不得标记为实现基线。对于员工 1 已批准的统一前端长期功能分支，项目负责人明确确认、里程碑冻结且本地门禁通过后，可以称为“正式里程碑基线”，供同一分支的后续本地开发继承；这不代表统一前端 P0 已最终交付。完成非作者评审、最终门禁并合入 `main`，仍是最终交付成立的条件。

## 安全与隐私

- 不在设计文件、Mock、截图或错误示例中放入 secret、真实账号或真实个人数据。
- 不把 Token、Cookie、密码、密钥或可猜测下载地址写入设计事实来源。
- 未确认的后端字段、错误结构和权限码不得出现在设计 Mock 中。

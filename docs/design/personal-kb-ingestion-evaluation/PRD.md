# 个人知识库、上传切分与检索评测 PRD

## 背景

当前企业知识库的上传入口位于用户工作区，普通用户没有可写空间；文档切分参数绑定在知识库创建阶段，无法按文档选择策略；检索测试集直接保存索引世代相关的 chunk ID，文档重新处理后会导致指标全部归零。

## 目标

1. 用户工作区的企业知识库只读，不显示上传入口。
2. 每个用户拥有且只能访问自己的个人知识库，可上传文档并用于 RAG 问答。
3. 文档统一转为经过清洗的 Markdown，再按上传时选择的固定长度、语义、递归或格式策略切分。
4. 检索评测在文档重建索引后仍能识别可精确映射的历史标注；不可精确映射时阻止生成误导性指标。
5. 在不引入新模型和复杂基础设施的前提下，提高检索准确性并保持可接受响应速度。

## 验收标准

- 企业知识库详情页无上传按钮和拖拽区；管理端仍可向企业知识库上传文档。
- 所有已有用户迁移后拥有一个个人知识库；注册和管理端新建用户时同步创建。
- 普通用户可向自己的个人知识库上传，不可向企业知识库或他人个人知识库上传。
- 上传请求包含 `chunk_strategy`、`chunk_size`、`chunk_overlap`，文档记录保存最终策略。
- 四种策略均有单元测试；Markdown 清洗仍在切分前完成。
- 历史测试集的失效标注可按同文档、同内容精确迁移；无法迁移时返回明确错误。
- 阈值在重排后基于重排分数过滤；未重排时优先使用向量分数。
- 前后端类型检查、Lint、相关测试和生产构建通过；Docker 部署健康。

## 非目标

- 不引入新的向量数据库或独立语义切分模型。
- 不自动用当前检索结果充当标准答案，以免评测数据泄漏。
- 不允许个人知识库跨用户共享。

## 风险

- 新切分策略会产生新的 chunk ID，旧测试集必须迁移标注。
- 语义切分采用句段边界启发式，不额外调用 LLM，质量与成本之间取保守平衡。
- 个人知识库涉及权限边界，后端所有读取、上传、检索都必须按所有者校验，不能只依赖前端隐藏。

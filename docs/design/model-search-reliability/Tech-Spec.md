# 模型管理与 AI 搜索可靠性修复技术方案

## 模型事务响应

- 模型或供应商写入并完成审计提交后，显式执行 `AsyncSession.refresh()`。
- 这样可重新加载由数据库生成的 `created_at`、`updated_at`，避免异步响应序列化触发隐式 I/O。
- 删除接口不刷新实体，保持原有响应契约。

## 模型真实测试

- `chat`：调用 `/chat/completions`，使用最少 512 个输出 Token，并要求存在非空可见回答。
- `embedding`：调用 `/embeddings`，要求返回一条非空向量；配置了维度时必须完全一致。
- `rerank`：调用供应商重排端点，要求返回至少一条合法排序结果。
- 失败响应只暴露状态类别和可操作提示，不透传供应商正文、密钥或内部堆栈。
- 前端成功文案改为“模型真实调用测试成功”，避免把供应商可访问误报为模型可用。

## 搜索前置校验

- 新增纯函数 `classifyUnsafeQuery()`，使用 NFKC、Unicode 字母数字保留规则，与后端确定性关键词规则保持一致。
- `AiSearchBox` 根据校验结果设置 `aria-invalid`、内联告警和提交按钮禁用状态。
- `SearchView.executeSearch()` 再做一次同源校验，覆盖快捷问题、历史状态或其他组件直接调用。
- 后端返回输入类 422 时，搜索页回到空闲状态并显示警告，不渲染检索故障面板。

## 验证

1. 后端单元测试覆盖写接口刷新顺序和三类模型实际调用。
2. 前端组件测试覆盖敏感词禁用、不跳转、不发 API，以及合法医疗词通过。
3. 运行 Ruff、Mypy、Pytest、Vue TypeScript、ESLint、Vitest 和生产构建。
4. 重建 API 与 Web 镜像，验证模型保存、模型真实测试和 AI 搜索正常问题。

## Docker 构建效率

- `.dockerignore` 必须使用 `**/.venv/` 等递归规则排除子目录虚拟环境和工具缓存，禁止把宿主机 Windows Python 环境复制进 Linux 镜像。
- API 与 Worker 继续使用非 root 用户；只把业务存储目录设为可写，应用代码与依赖保持只读。
- 禁止对包含 PyTorch 的整个 `/app` 或 `/app/backend` 递归修改所有权，否则每次源码变更都会产生数分钟的无效镜像层操作。

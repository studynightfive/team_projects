# 模型管理与 AI 搜索可靠性修复 PRD

## 背景

真实环境中编辑模型配置时，业务事务已经提交，但接口序列化服务端更新时间字段时触发 SQLAlchemy `MissingGreenlet`，前端收到 500。AI 搜索对明确敏感词的拦截发生在跳转结果页之后，用户只能看到检索失败；模型管理的连通性测试又只验证供应商 `/models` 接口，无法证明所选聊天、向量或重排模型可以实际调用。

## 目标

1. 创建和编辑模型、供应商后稳定返回最新数据，不出现提交成功但响应失败。
2. 模型测试真实调用所选模型，并对空回答、向量维度不一致和重排无结果给出安全、明确的提示。
3. 首页和搜索页输入明确敏感词时立即显示“输入内容不合法，请修改后重试”，禁用搜索按钮且不发起检索。
4. 后端输入守卫继续作为不可绕过的最终安全边界。
5. 当前启用的聊天、Embedding 和 Rerank 配置均可被 RAG 链路使用。

## 验收标准

- `PATCH /api/v1/models/{id}` 返回 200，响应包含可读取的 `updated_at`。
- 模型测试按 `chat`、`embedding`、`rerank` 类型调用对应业务端点，不再仅凭 `/models` 返回 200 判定成功。
- 推理型聊天模型的测试至少提供 512 个输出 Token，空可见内容判定为失败。
- 输入“如何贩毒”等敏感内容时，按钮不可点击、输入框标记无效、页面不跳转、业务 API 不调用。
- “医院病毒防控系统如何建设”等合法医疗问题不被关键词前置校验误伤。
- 直接绕过前端提交敏感内容时，FastAPI 仍返回 422。
- 前后端类型、Lint、单元测试和生产构建通过。

## 约束与风险

- 前端敏感词判断只用于即时交互，不能替代后端确定性关键词校验。
- 后端不再执行概率模型守卫；提示词注入与复杂语义主题需由供应商内容安全、权限和审计
  等措施补充治理。
- 真实模型测试会产生一次最小供应商调用，页面必须明确这是模型调用测试。
- 不读取、回显或改写任何现有 API Key。

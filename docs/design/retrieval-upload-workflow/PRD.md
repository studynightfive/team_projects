# 检索与上传工作流优化 PRD

## 1. 背景

当前上传、下载任务只有在页面重新挂载时才刷新，AI 搜索只能选择一个知识库，
答案缓存主要依赖问题文本近似度，文档上传也未对同一知识库内的同名文档提供明确处理流程。
这些问题会让用户误以为任务停止、降低跨库检索效率，并可能产生不可区分的文档。

## 2. 目标

1. 上传、下载及后台文档任务在当前页面自动刷新状态和进度，不依赖刷新或切换页面。
2. 相同语义的问题能够命中答案缓存，不同意图、否定关系或关键实体不同的问题不得误命中。
3. AI 搜索可同时选择 1 至 10 个知识库，并阻止选择同名知识库。
4. 同一知识库内文档名称唯一；发生冲突时，用户必须选择替换已有文档或添加标识后保留两份。

## 3. 用户流程

### 3.1 任务进度

- 用户确认上传后立即看到每个文档的处理阶段和进度条。
- 下载中心和管理端任务列表在存在进行中任务时自动刷新。
- 页面离开后停止轮询，终态任务不再产生额外请求。

### 3.2 多知识库检索

- 搜索框默认选中首个可访问知识库。
- 用户点击知识库选择区旁的加号添加更多知识库。
- 已选知识库可单独移除；至少选择一个知识库后才能搜索。
- 名称忽略首尾空格和大小写后相同的知识库不能同时选择。

### 3.3 同名文档

- 上传前检查文件名对应的文档名称是否已存在。
- 有冲突时暂停上传，列出冲突文档，并要求选择：
  - 替换已有文档；
  - 自动添加序号标识后作为新文档上传；
  - 取消本次上传。
- 即使绕过前端，后端和数据库也必须阻止未声明策略的同名写入。

## 4. 验收标准

- 上传后进度条每秒自动变化，完成后文档列表自动刷新。
- 下载中心和管理端任务列表无需刷新即可进入终态。
- 语义相同但表述不同的问题可命中缓存，返回缓存相似度和语义命中类型。
- 否定关系、问题类型、数字或英文关键实体不同的问题不会命中语义缓存。
- 一次搜索可选择多个不同名称的知识库，后端仅在这些知识库范围检索。
- 同名知识库无法重复加入检索范围，直接调用接口也会被拒绝。
- 同一知识库中不能存在两个活动状态的同名文档。
- 文档冲突的“替换”和“添加标识”两种处理均有自动化测试。

## 5. 约束与风险

- 保留单知识库 `kb_id` 以兼容旧客户端，新前端只发送 `kb_ids`。
- 语义缓存仅复用相同用户、知识库范围、模型、知识版本和检索参数下的答案。
- 首次语义缓存判断需要一次 Embedding 调用；该向量同时用于后续检索，不重复请求模型。
- 数据库迁移会为历史同名文档添加稳定标识，避免唯一索引创建失败。

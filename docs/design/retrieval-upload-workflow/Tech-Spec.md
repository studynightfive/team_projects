# 检索与上传工作流技术方案

## 1. 接口契约

### 1.1 多知识库检索

`SearchRequest` 新增：

```json
{
  "kb_ids": ["kb-a", "kb-b"]
}
```

- `kb_ids` 最多 10 个，自动去重前先拒绝重复 ID。
- `kb_id` 保留兼容，但不得与 `kb_ids` 同时传入。
- 后端校验全部 ID 均在当前用户可访问范围内。
- 后端校验所选知识库名称在标准化后唯一。

### 1.2 文档名称冲突预检

新增：

```text
POST /api/v1/knowledge-bases/{kb_id}/documents/name-conflicts
```

请求：

```json
{"filenames": ["病历规范.pdf", "接口标准.docx"]}
```

响应返回冲突文件名、标准化文档名称、已有文档 ID 和显示名称。

上传表单 `duplicate_policy` 支持：

- `replace`：替换同名活动文档；
- `rename`：在名称和文件名后添加递增标识；
- `new_version`：仅兼容无名称冲突的旧流程，遇到同名时返回 409。

## 2. 语义缓存

缓存条目保存查询向量及查询意图特征。命中流程：

1. 先按标准化问题读取精确缓存，不调用 Embedding。
2. 精确未命中时生成一次查询向量。
3. 在同一缓存作用域候选中计算余弦相似度。
4. 相似度达到 `0.95`，且问题类型、否定极性、数字和英文关键实体兼容时才复用。
5. 未命中时同一查询向量传给向量或混合检索，避免重复 Embedding。

旧版本无向量的缓存条目会被安全忽略，缓存键版本升级后自然淘汰。

## 3. 进度刷新

- 上传接口返回任务 ID，前端转为统一任务进度项并使用现有任务查询接口每秒轮询。
- 下载中心和管理端任务列表仅在存在 `pending`、`queued`、`running` 等非终态记录时轮询。
- 使用 `setTimeout` 串行调度，避免上一次请求未结束时叠加。
- 页面卸载时清除定时器并中止请求；网络错误后继续重试并保留上次状态。

## 4. 数据一致性

- 活动文档对 `(knowledge_base_id, lower(trim(title)))` 建立部分唯一索引。
- 上传冲突检查和最终写入均在服务层执行，数据库索引处理并发竞争。
- 历史重复名称在迁移时追加文档 ID 标识，保留全部原始数据。

## 5. 验证

- 后端：Schema、缓存意图保护、多知识库权限、同名预检、替换与重命名单元/集成测试。
- 前端：上传冲突交互、进度轮询、多知识库添加和移除、同名知识库阻止测试。
- 门禁：Ruff、Mypy、Pytest、Typecheck、ESLint、Vitest、生产构建。
- 视觉：1440px、1280px、375px 检查搜索选择器、冲突提示和进度面板。

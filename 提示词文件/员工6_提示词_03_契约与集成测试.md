# 员工6 提示词 03：契约与集成测试

> 角色：员工6（测试与平台工程）
> 对应分支：`feature/<issue>-contract-integration-tests`
> 对应周次：第2-7周（贯穿整个开发周期，跟随各模块接口就位逐步覆盖）
> 前置条件：提示词01中的 CI 和 Docker 环境已就位；提示词02中的样本库基础已建立

---

## 一、背景

方案第10.1节定义了接口协作顺序：后端先提交 OpenAPI 草案 -> 前端确认字段 -> 前端使用 Mock 开发 -> 后端实现完成 -> 你运行契约测试。方案第16.4节定义了测试层级，你负责契约测试、集成测试、性能测试和安全测试的协作。

你的职责是：确保 OpenAPI 是接口事实来源，前端类型和 Mock 与后端实现一致，核心业务流程端到端可工作，越权、上传、XSS、Secret 和下载安全经过测试验证。

---

## 二、目标

1. 契约测试验证 OpenAPI 与前端类型、Mock 和后端实现一致。
2. 集成测试覆盖核心业务流程：上传到检索、问答到引用、导出到下载。
3. 权限测试矩阵覆盖所有场景。
4. 安全测试覆盖越权、上传、XSS、Secret 泄露和下载安全。
5. 性能测试覆盖大文件、并发问答、检索延迟和队列积压。

---

## 三、执行步骤

### 步骤 1：建立契约测试框架

**文件路径**：`backend/tests/contract/`

**框架要求**：

1. 创建 `backend/tests/contract/conftest.py`，提供共享 fixture：
   - `client`：FastAPI TestClient（或 httpx.AsyncClient）。
   - `auth_headers(user_role)`：根据角色返回认证头。
   - `admin_headers`：完整管理员认证头。
   - `partial_admin_headers`：部分权限管理员认证头。
   - `user_headers`：普通用户认证头。

2. 创建 `backend/tests/contract/test_openapi_consistency.py`：
   - 从 `docs/api/openapi.yaml` 加载 OpenAPI 规范。
   - 对每个端点，验证请求和响应结构与 OpenAPI 定义一致。
   - 验证必填字段存在、可选字段类型正确。
   - 验证错误响应结构与统一响应格式一致。

3. 创建 `backend/tests/contract/test_response_format.py`：
   - 所有非 SSE 端点返回统一响应格式 `{"code": 0, "message": "success", "data": ..., "request_id": "uuid"}`。
   - 分页端点返回 `{"code": 0, "message": "success", "data": {"items": [], "page": 1, "page_size": 20, "total": 0}, "request_id": "uuid"}`。
   - 每个响应包含 `request_id` 字段。
   - 时间字段使用 ISO 8601 格式。

验收标准：
- 契约测试框架就位，conftest.py 提供认证 fixture。
- `test_openapi_consistency.py` 能加载 OpenAPI 规范并验证端点。
- `test_response_format.py` 验证统一响应格式。
- 测试框架集成到 CI 的 `quality` job 中。

### 步骤 3：编写权限测试矩阵

**文件路径**：`backend/tests/contract/test_permission_matrix.py`

**测试场景**（来自方案第16.5节）：

| 场景 | 预期 | 测试方法 |
|---|---|---|
| 普通用户登录 | 不显示管理中心 | 验证 `/me` 返回的 permissions 不包含 `admin.*` |
| 普通用户请求 `/admin` 接口 | 返回 403 | 对所有管理接口发送普通用户请求 |
| 仅用户查看权限的管理员 | 只显示用户管理查看功能 | 创建仅有 `admin.user.view` 的角色，验证只能访问用户列表 |
| 部分管理员 | 只显示获授权模块和操作 | 创建仅有部分权限的角色，逐个验证未授权接口返回 403 |
| 完整管理员 | 所有管理接口可用 | 创建有全部管理权限的角色，验证所有接口 |
| 权限被撤销 | 刷新权限并退出失权页面 | 修改用户权限后，验证 `/me` 返回更新后的权限 |
| 后端返回 403 | 前端不保留伪成功状态 | 验证 403 响应不包含敏感数据 |
| 用户被禁用 | 会话按契约失效 | 禁用用户后，使用该用户的 Token 请求应返回 401 |
| 知识库数据权限 | 只能看到有权知识库 | 创建不同知识库授权的用户，验证文档列表和检索结果 |
| 文档资源权限 | 无权文档不可访问 | 验证无权文档的预览、下载和检索结果 |
| 批量接口权限 | 逐项或整体校验 | 批量操作中包含无权资源时，验证处理策略 |

**测试数据要求**：
- 测试数据库预置多个角色：普通用户、仅查看管理员、部分管理员、完整管理员。
- 预置多个知识库，每个知识库授权给不同用户。
- 预置多个文档，每个文档属于不同知识库。
- 测试数据在每次测试前重置。

验收标准：
- 所有16.5节列出的权限场景有对应测试。
- 权限测试通过 CI。
- 管理接口权限校验 100%（前端隐藏不能替代后端校验）。

### 步骤 4：编写集成测试

**文件路径**：`backend/tests/integration/`

**核心业务流程覆盖**：

**A. 上传到检索流程**（`test_upload_to_retrieval.py`）：
1. 管理员上传一个文档（使用样本库中的 PDF 样本）。
2. 等待文档处理完成（状态变为 `ready`）。
3. 使用关键词检索验证文档可被搜索到。
4. 使用向量检索验证文档可被搜索到。
5. 使用混合检索验证文档可被搜索到。
6. 验证检索结果包含正确的来源、页码、片段和分数。

**B. 问答到引用流程**（`test_chat_to_citation.py`）：
1. 确保知识库中有已索引的文档。
2. 发起流式问答请求。
3. 验证 SSE 事件序列正确：`start` -> `delta`* -> `citation`* -> `done`。
4. 验证引用包含 `citation_id`、`document_id`、`document_name`、`page_no`、`quote`、`score`。
5. 验证引用关联的文档用户有权访问。
6. 验证会话和消息已保存。

**C. 导出到下载流程**（`test_export_to_download.py`）：
1. 用户选择文档和格式发起导出。
2. 等待导出任务完成。
3. 验证导出文件存在且大小大于 0。
4. 鉴权下载导出文件。
5. 验证文件内容与选择范围一致。
6. 验证无权用户不能下载他人的导出文件。
7. 验证导出文件过期后无法下载。

**D. 文档生命周期流程**（`test_document_lifecycle.py`）：
1. 管理员上传文档，验证状态从 `uploaded` 流转到 `ready`。
2. 预览文档 Markdown 和资源。
3. 删除文档，验证状态变为 `deleted` 或物理删除。
4. 验证删除后的文档不出现在检索结果中。
5. 验证历史引用中的已删除文档显示不可访问状态。

**E. 会话管理流程**（`test_conversation_lifecycle.py`）：
1. 创建会话并发起问答。
2. 重命名会话，验证名称已更新。
3. 搜索会话，验证可搜索到。
4. 删除会话，验证会话不再出现在列表中。
5. 验证会话中的消息和引用仍然保留（审计用途）。

**F. 权限变更流程**（`test_permission_change.py`）：
1. 用户 A 有权访问知识库 K1。
2. 用户 A 发起问答，引用 K1 中的文档。
3. 管理员撤销用户 A 对 K1 的权限。
4. 用户 A 刷新权限，验证 K1 不再出现在可用知识库列表中。
5. 用户 A 查看历史引用，验证引用显示不可访问状态，不泄露原文。

验收标准：
- 6 个核心业务流程集成测试通过。
- 集成测试使用 Docker Compose 测试环境运行（`docker-compose.test.yml`）。
- 集成测试在 CI 中运行（允许比其他测试更长的超时时间）。

### 步骤 5：编写安全测试

**文件路径**：`backend/tests/security/`

**测试覆盖**：

**A. 越权测试**（`test_authorization.py`）：
- 普通用户尝试访问管理接口 -> 403。
- 用户 A 尝试访问用户 B 的会话 -> 403。
- 用户 A 尝试下载用户 B 的导出文件 -> 403。
- 用户 A 尝试通过修改 UUID 访问无权知识库 -> 403 或 404。
- 用户 A 尝试通过修改 UUID 查看无权文档 -> 403 或 404。
- 部分管理员尝试访问未授权模块 -> 403。

**B. 上传安全测试**（`test_upload_security.py`）：
- 上传文件名包含 `../` 路径穿越尝试 -> 被拒绝或清理。
- 上传文件扩展名与实际内容不匹配 -> 被检测并拒绝。
- 上传超大文件 -> 被拒绝并返回明确错误。
- 上传空文件 -> 被拒绝或正确处理。
- 同时上传大量文件 -> 不导致服务崩溃。

**C. XSS 测试**（`test_xss.py`）：
- 上传包含 `<script>alert('xss')</script>` 的 Markdown 文件 -> 渲染后脚本被移除。
- 上传包含 `onclick` 事件属性的 HTML 文件 -> 事件属性被移除。
- 上传包含 `javascript:` 链接的 HTML 文件 -> 危险协议被移除。
- 模型输出包含 `<img onerror=...>` 的假设场景 -> 前端 DOMPurify 过滤。

**D. Secret 泄露测试**（`test_secret_leak.py`）：
- 模型密钥 GET 接口 -> 不返回明文或可逆内容。
- 错误响应中 -> 不包含数据库连接字符串。
- 日志输出中 -> 不包含 Authorization 头。
- 导出文件的元数据中 -> 不包含内部路径。
- 公开错误响应中 -> 不暴露存储路径、SQL、模型密钥或内部地址。

**E. 下载安全测试**（`test_download_security.py`）：
- 直接访问导出文件存储路径 -> 被拒绝。
- 使用过期的下载链接 -> 被拒绝。
- 使用他人的下载链接 -> 被拒绝。
- 猜测下载 URL -> 无法访问（不暴露可猜测静态地址）。
- 下载链接不包含 Token 在 URL 中。

验收标准：
- 所有 5 类安全测试通过。
- 严重安全问题为 0（越权、Secret 泄露、任意文件访问、存储型 XSS）。
- 安全测试在 CI 中运行。

### 步骤 6：编写性能测试

**文件路径**：`backend/tests/performance/`

**测试场景**（不要求每次 CI 运行，但必须可手动或定期执行）：

**A. 大文件处理测试**（`test_large_file.py`）：
- 上传 50MB 的 PDF 文件 -> 处理完成，不超时，不导致内存溢出。
- 上传 100MB 的 DOCX 文件 -> 处理完成或被合理拒绝。

**B. 并发问答测试**（`test_concurrent_chat.py`）：
- 同时 10 个用户发起流式问答 -> 所有请求正常处理，不出现 503。
- 同时 50 个用户发起流式问答 -> 系统不崩溃，队列正常排队。

**C. 检索延迟测试**（`test_retrieval_latency.py`）：
- 在 1000 个文档的知识库中检索 -> P95 延迟不超过 2 秒。
- 在 10000 个文档的知识库中检索 -> P95 延迟不超过 5 秒。

**D. 队列积压测试**（`test_queue_backlog.py`）：
- 同时上传 20 个文档 -> 全部处理完成，队列不阻塞。
- 上传处理期间停止 Worker -> 任务保留在队列中，Worker 恢复后继续处理。

**性能测试报告**：
- 每次运行记录性能数据（延迟、吞吐量、错误率）。
- 与基线对比，性能退化超过 20% 时标记为失败。

验收标准：
- 性能测试脚本就位，可手动或定期执行。
- 性能基线已记录。
- 性能退化超过阈值时 CI 发出警告。

1. 测试数据不包含真实用户信息、密码、Token 或商业 Secret。
2. 测试数据可重复生成（通过 fixture 或 seed 脚本）。

4. 测试数据在 CI 环境中自动准备。

- 预置测试用户：`admin`（完整管理员）、`partial_admin`（部分管理员）、`viewer_admin`（仅查看管理员）、`normal_user`（普通用户）、`disabled_user`（已禁用用户）。

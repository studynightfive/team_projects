## 三、执行步骤

### 步骤 1：建立结构化日志体系

**技术选型**：`structlog`（方案第13.3节已锁定版本 25.4.0）。

**文件路径**：`backend/app/common/logging.py`

**要求**：

1. 创建统一的日志配置模块，供 API Server 和 Worker 共用。

2. 日志格式（JSON 格式，便于采集和搜索）：
   ```json
   {
     "timestamp": "2026-07-15T08:00:00.000Z",
     "level": "info",
     "request_id": "uuid",
     "user_id": "uuid",
     "action": "document.upload",
     "resource_type": "document",
     "resource_id": "uuid",
     "result": "success",
     "duration_ms": 150,
     "message": "Document uploaded successfully"
   }
   ```

3. 每个请求必须记录：
   - `request_id`：从请求头或中间件生成，贯穿整个请求生命周期。
   - `user_id`：当前认证用户（如有）。
   - `action`：操作类型（如 `auth.login`、`document.upload`、`chat.stream`）。
   - `resource_type` 和 `resource_id`：操作的目标资源。
   - `result`：`success`、`failure`、`denied`。
   - `duration_ms`：请求处理耗时。
   - 错误日志额外包含 `error_code`、`error_message`（仅安全信息，不包含堆栈）。

4. 日志绝对不能包含：
   - Authorization 头或 Cookie 值。
   - 密码或密码哈希。
   - 模型密钥或 API Key。
   - 完整的请求体或响应体（仅记录必要字段）。
   - 内部存储路径、SQL 语句或数据库连接字符串。

5. 日志级别使用：
   - `INFO`：正常请求和操作。
   - `WARNING`：权限拒绝、资源不存在、频率限制触发。
   - `ERROR`：异常和失败（仅安全错误信息）。
   - `DEBUG`：详细诊断信息（仅开发环境，生产环境关闭）。

6. 中间件实现：
   - 在 FastAPI 中间件中自动注入 `request_id`。
   - 在中间件中自动记录请求开始和结束，计算 `duration_ms`。
   - 异常处理器中自动记录错误日志。

验收标准：
- 所有 API 请求自动记录结构化日志。
- 日志不包含任何 Secret 或敏感信息。
- 日志格式为 JSON，便于日志采集系统解析。
- Worker 任务的日志同样使用结构化格式。

### 步骤 2：建立 Prometheus 指标

**技术选型**：`prometheus-client`（方案第13.3节已锁定版本 0.23.1）。

**文件路径**：`backend/app/common/metrics.py`

**指标端点**：`GET /api/v1/metrics`（仅内部网络可访问，或通过 Nginx 限制）。

**必须实现的指标**：

| 指标名称 | 类型 | 标签 | 说明 |
|---|---|---|---|
| `api_requests_total` | Counter | method, endpoint, status_code | API 请求总量 |
| `api_request_duration_seconds` | Histogram | method, endpoint | API 请求延迟分布 |
| `sse_connections_active` | Gauge | - | 当前活跃 SSE 连接数 |
| `sse_connections_total` | Counter | status | SSE 连接总数（按 completed/failed/cancelled 分类） |
| `document_tasks_total` | Counter | status, stage | 文档任务总数（按状态和阶段分类） |
| `document_task_queue_length` | Gauge | - | 文档任务队列当前长度 |
| `document_task_wait_seconds` | Histogram | - | 文档任务等待时间分布 |
| `document_task_duration_seconds` | Histogram | stage | 文档各阶段处理耗时（OCR、转换、索引等） |
| `retrieval_requests_total` | Counter | mode, status | 检索请求总数（按关键词/向量/混合和状态分类） |
| `retrieval_latency_seconds` | Histogram | mode | 检索延迟分布 |
| `retrieval_no_hits_total` | Counter | mode | 无命中检索次数 |
| `model_requests_total` | Counter | model_id, status | 模型请求总数 |
| `model_request_duration_seconds` | Histogram | model_id | 模型请求耗时分布 |
| `model_tokens_total` | Counter | model_id, type | Token 使用量（按 input/output 分类） |
| `export_tasks_total` | Counter | format, status | 导出任务总数 |
| `export_task_duration_seconds` | Histogram | format | 导出任务耗时分布 |
| `db_connections_active` | Gauge | - | 活跃数据库连接数 |
| `db_query_duration_seconds` | Histogram | - | 慢查询耗时分布 |
| `redis_connected_clients` | Gauge | - | Redis 连接数 |
| `redis_memory_used_bytes` | Gauge | - | Redis 内存使用 |
| `file_storage_used_bytes` | Gauge | - | 文件存储使用量 |

**指标实现要求**：
- 使用 `prometheus_client` 的 `Counter`、`Gauge`、`Histogram`。
- Histogram 的 bucket 根据实际场景设置合理范围（如 API 延迟：0.01, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10）。
- 指标在中间件中自动收集，不要求每个端点手动埋点。
- 特殊指标（如 SSE 连接数、任务队列长度）在对应模块中手动更新。

**Prometheus 配置**（`deploy/prometheus/prometheus.yml`）：
- 抓取间隔：15 秒。
- 目标：`api-server:8000`。
- 可选：如果使用 Grafana，提供基础仪表板 JSON 模板。

验收标准：
- `/api/v1/metrics` 端点返回 Prometheus 格式的指标数据。
- 所有指标在请求处理中自动更新。
- 指标端点受 Nginx 或内部网络限制，不对外暴露。

### 步骤 3：建立健康检查端点

**文件路径**：`backend/app/api/health.py`

**端点**：

**A. 存活检查**：`GET /api/v1/health/live`
- 返回 200，表示进程存活。
- 响应：`{"status": "ok", "timestamp": "..."}`

**B. 就绪检查**：`GET /api/v1/health/ready`
- 检查数据库连接：执行 `SELECT 1`。
- 检查 Redis 连接：执行 `PING`。
- 检查文件存储：检查存储目录是否可读写。
- 全部通过返回 200，任一失败返回 503。
- 响应：
  ```json
  {
    "status": "ok",
    "checks": {
      "database": "ok",
      "redis": "ok",
      "storage": "ok"
    },
    "timestamp": "..."
  }
  ```

**C. Worker 健康检查**：
- Worker 进程定时向 Redis 写入心跳键。
- 健康检查端点读取心跳键，判断 Worker 是否在阈值时间内更新过心跳。
- 心跳超时阈值：30 秒。

**D. 完整健康检查脚本**：`scripts/health-check.sh`

```bash
#!/bin/bash
# 检查所有服务健康状态

echo "=== Web ==="
curl -s -o /dev/null -w "%{http_code}" http://localhost:80/

echo "=== API Server Live ==="
curl -s http://localhost:80/api/v1/health/live

echo "=== API Server Ready ==="
curl -s http://localhost:80/api/v1/health/ready

echo "=== PostgreSQL ==="
docker compose -f deploy/docker-compose.yml exec postgres pg_isready -U postgres

echo "=== Redis ==="
docker compose -f deploy/docker-compose.yml exec redis redis-cli ping
```

验收标准：
- 所有5个服务（web、api-server、worker、postgres、redis）有健康检查。
- `docker compose` 的 `depends_on` 使用 `condition: service_healthy`。
- 健康检查脚本可一键检查全部服务状态。
- 健康检查不暴露内部细节（如数据库连接字符串）。

### 步骤 5：建立 PostgreSQL 备份方案

**文件路径**：`scripts/backup-postgres.sh`

**备份策略**：
- 每天全量逻辑备份（`pg_dump`）。
- 保留最近 7 天的每日备份。
- 保留最近 4 周的每周备份。
- 保留最近 3 个月的每月备份。

**脚本要求**：
```bash
#!/bin/bash
# PostgreSQL 备份脚本
# 用法: bash scripts/backup-postgres.sh [backup_dir]

set -euo pipefail

BACKUP_DIR="${1:-./backups/postgres}"
TIMESTAMP=$(date -u +"%Y%m%d_%H%M%S")
BACKUP_FILE="${BACKUP_DIR}/knowledge_base_${TIMESTAMP}.sql.gz"

# 创建备份目录
mkdir -p "${BACKUP_DIR}"

# 执行备份
docker compose -f deploy/docker-compose.yml exec -T postgres \
  pg_dump -U postgres knowledge_base | gzip > "${BACKUP_FILE}"

# 验证备份文件完整性
if gzip -t "${BACKUP_FILE}"; then
  echo "Backup successful: ${BACKUP_FILE}"
else
  echo "Backup failed: file is corrupted"
  rm -f "${BACKUP_FILE}"
  exit 1
fi

# 清理过期备份
# 删除7天前的每日备份
find "${BACKUP_DIR}" -name "knowledge_base_*.sql.gz" -mtime +7 -delete

echo "Backup completed at $(date)"
```

**关键要求**：
- 备份文件使用 gzip 压缩。
- 备份后验证文件完整性。
- 备份文件不包含 Secret 明文（数据库连接密码通过环境变量或 `.pgpass` 文件提供）。
- 备份文件名包含时间戳。
- 备份脚本退出码正确（成功 0，失败 1）。

验收标准：
- 备份脚本可执行，生成有效的 `.sql.gz` 文件。
- 备份文件可通过 `gzip -t` 完整性检查。
- 过期备份自动清理。
- 备份脚本集成到 CI 或 cron 中。

### 步骤 6：建立文件存储备份方案

**文件路径**：`scripts/backup-files.sh`

**备份范围**：
- `storage/documents/`：原始文件、Markdown、assets、manifest。
- 不备份向量索引（可通过文档重建）。

**备份策略**：
- 每天增量备份（使用 `rsync`）。
- 每周全量备份。
- 保留策略与 PostgreSQL 备份一致。

**脚本要求**：
```bash
#!/bin/bash
# 文件存储备份脚本
# 用法: bash scripts/backup-files.sh [backup_dir]

set -euo pipefail

BACKUP_DIR="${1:-./backups/files}"
TIMESTAMP=$(date -u +"%Y%m%d_%H%M%S")
STORAGE_DIR="./storage"

# 使用 rsync 进行增量备份
rsync -av --delete "${STORAGE_DIR}/" "${BACKUP_DIR}/current/"

# 创建带时间戳的归档
tar -czf "${BACKUP_DIR}/files_${TIMESTAMP}.tar.gz" -C "${BACKUP_DIR}/current" .

echo "File backup completed at $(date)"
```

**关键要求**：
- 备份包含原始文件、Markdown、assets 和 manifest。
- 同一恢复点策略：PostgreSQL 备份和文件存储备份的时间戳对齐，确保恢复时数据一致。
- 备份脚本退出码正确。

验收标准：
- 备份脚本可执行，生成有效的 `.tar.gz` 文件。
- 备份文件包含 `storage/` 下的所有内容。
- 备份与 PostgreSQL 备份时间戳对齐。

### 步骤 7：建立恢复方案

**文件路径**：`scripts/restore.sh`

**恢复脚本要求**：
```bash
#!/bin/bash
# 数据库和文件存储恢复脚本
# 用法: bash scripts/restore.sh <postgres_backup_file> <files_backup_file>

set -euo pipefail

POSTGRES_BACKUP="${1}"
FILES_BACKUP="${2}"

# 1. 恢复 PostgreSQL
echo "Restoring PostgreSQL..."
gunzip -c "${POSTGRES_BACKUP}" | \
  docker compose -f deploy/docker-compose.yml exec -T postgres \
  psql -U postgres knowledge_base

# 2. 恢复文件存储
echo "Restoring file storage..."
tar -xzf "${FILES_BACKUP}" -C ./storage/

# 3. 重建向量索引（如果需要）
echo "Reindexing vectors..."
# 触发向量索引重建（通过 API 或管理命令）
# docker compose exec api-server uv run python -m backend.scripts.reindex_vectors

# 4. 验证恢复
echo "Verifying restoration..."
# 检查关键表行数
docker compose -f deploy/docker-compose.yml exec -T postgres \
  psql -U postgres -d knowledge_base -c "SELECT count(*) FROM users;"
docker compose -f deploy/docker-compose.yml exec -T postgres \
  psql -U postgres -d knowledge_base -c "SELECT count(*) FROM documents;"

echo "Restore completed at $(date)"
```

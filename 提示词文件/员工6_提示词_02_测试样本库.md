# 员工6 提示词 02：测试样本库

> 角色：员工6（测试与平台工程）
> 对应分支：`feature/<issue>-test-samples`
> 对应周次：第2-3周（持续积累到第8周）
> 前置条件：提示词01中的 CI 和 Docker 环境已就位；员工4开始提交文档处理代码

---

## 一、背景

方案第16.6节要求为每个支持的文档格式至少准备正常样本、异常样本、中文内容样本和含表格/图片/页码的代表性样本。方案第1.3节要求每类至少1个正常样本和1个异常样本。样本库是文档处理、OCR、检索和导出功能测试的基础，也是 CI 回归测试的数据来源。

你的职责是：建立样本库的目录结构、收集和制作样本文件、编写样本清单和验证脚本，并确保样本不包含真实个人信息和商业 Secret。

---

## 二、目标

1. 为方案第6.2节列出的所有格式准备正常和异常样本。
2. 为 OCR 场景准备图片和扫描 PDF 样本。
3. 为五种导出格式准备回归验证样本。
4. 样本库结构清晰，每个样本有元数据说明。
5. 样本不包含任何真实敏感数据。

## 四、执行步骤

### 步骤 1：创建样本库骨架和说明文档

**文件路径**：`samples/README.md`

**内容要求**：
- 样本库用途说明。
- 目录结构说明。
- 样本命名规范。
- 如何添加新样本。
- 样本内容限制（禁止真实个人信息、商业 Secret、受版权保护内容）。
- 每个样本的元数据说明。

**文件路径**：`samples/manifest.json`

**格式要求**（机器可读的样本清单）：

```json
{
  "version": "1.0",
  "samples": [
    {
      "id": "pdf-001",
      "path": "documents/pdf/normal/simple-text.pdf",
      "format": "pdf",
      "category": "normal",
      "language": "zh-CN",
      "features": ["text", "paragraphs"],
      "page_count": 3,
      "file_size_bytes": 0,
      "sha256": "",
      "description": "纯文本中文PDF，3页，含标题和正文段落"
    }
  ]
}
```

验收标准：
- `samples/README.md` 包含完整的目录结构说明和命名规范。
- `samples/manifest.json` 格式正确，作为样本清单模板。

### 步骤 2：收集和制作文档样本

**正常样本制作要求**（按方案第6.2节的格式表）：

对每一类格式，创建至少1个正常样本：

**PDF 类**：
- 使用 LibreOffice 或任意工具生成含中文标题、正文段落、表格和图片的 PDF。
- 扫描件：打印一份中文文档后扫描为 PDF（如无扫描仪，使用手机拍照后转换为 PDF）。

**Office 类（Word、Excel、PowerPoint）**：
- 使用 LibreOffice 或 Microsoft Office 创建。
- 内容为假数据（如"项目名称：示例项目"、"联系人：张三"、"总计：1000"），不使用真实数据。
- Excel 包含合并单元格、多工作表样本。
- PowerPoint 包含标题、正文、备注和图片。

**Markdown 和文本类**：
- 使用文本编辑器直接创建。
- 包含中文内容、代码块、表格、图片引用。
- 异常样本包含 `<script>alert('xss')</script>` 等危险内容（用于清洗测试）。

**HTML 类**：
- 正常样本：简单的文章页面，含标题、段落、表格。
- 异常样本：含 `<script>`、`onclick`、`javascript:` 链接等危险内容。

**图片类**：
- 使用截图工具截取中文文档、英文文档、表格等内容。
- 异常样本：缩小到极低分辨率、旋转90度、纯图片无文字。

**邮件类**：
- 使用邮件客户端导出一封简单邮件为 `.eml` 格式。
- 内容为假数据。

**电子书类**：
- 使用 Calibre 或在线工具生成简单 EPUB。
- ODT/ODS/ODP 使用 LibreOffice 创建。

**核心原则**：
- 所有内容使用假数据，不包含任何真实个人信息。
- 不包含公司内部真实文档。
- 不包含受版权保护的内容。
- 文件大小控制在合理范围（单个文件不超过 10MB，扫描件除外）。

验收标准：
- 方案第6.2节列出的每种格式至少有1个正常样本。
- 样本内容为中文假数据。
- 样本不包含真实个人信息或商业 Secret。

### 步骤 3：收集和制作异常样本

对每一类格式，创建至少1个异常样本：

| 格式 | 异常样本类型 | 制作方式 |
|---|---|---|
| PDF | 损坏 PDF | 截断正常 PDF 文件的后半部分 |
| PDF | 加密 PDF | 使用 PDF 工具设置打开密码 |
| DOCX | 损坏 DOCX | 截断正常 DOCX 文件 |
| XLSX | 空工作簿 | 创建无任何内容的 XLSX |
| PPTX | 无内容 | 创建无任何内容的 PPTX |
| Markdown | 危险 HTML | 在 Markdown 中嵌入 `<script>` 和 `onerror` 等 |
| Markdown | 空文件 | 0 字节文件 |
| TXT | 二进制伪装 | 将二进制文件重命名为 `.txt` |
| CSV | 列数不一致 | 手动构造不同行不同列数的 CSV |
| CSV | 空文件 | 0 字节文件 |
| HTML | 含脚本 | 含 `<script>` 和事件属性的 HTML |
| HTML | 深度嵌套 | 100 层嵌套 div |
| EPUB | 损坏 | 移除 EPUB 内的关键文件后重新打包 |
| 图片 | 低分辨率 | 缩放到 50x50 像素 |
| 图片 | 旋转 | 旋转 90 度 |
| 图片 | 无文字 | 纯色块图片 |
| EML | 损坏 MIME | 截断 MIME 边界 |
| ODT | 空文件 | 0 字节或空内容 |

验收标准：
- 方案第6.2节列出的每种格式至少有1个异常样本。
- 异常样本能触发预期的错误处理路径（而非崩溃）。

### 步骤 4：创建 OCR 专项样本

**要求**：
- 至少 3 个不同清晰度的中文扫描件/图片。
- 至少 1 个英文扫描件/图片。
- 至少 1 个含表格的扫描件/图片。
- 至少 1 个低清晰度或旋转的图片（用于测试 OCR 鲁棒性）。

**样本内容**：
- 使用公开领域的文本（如政府公告、维基百科开放内容）生成扫描件。
- 内容包含中文、英文、数字和基本标点符号。
- 为每个样本准备预期文本（Ground Truth），用于评估 OCR 准确率。

**Ground Truth 文件**：`samples/documents/images/normal/ground-truth.json`

```json
{
  "clear-cn.png": {
    "expected_text": "这是预期的OCR识别结果文本...",
    "min_confidence": 0.7,
    "key_phrases": ["关键词1", "关键词2"]
  }
}
```

验收标准：
- 至少 3 个中文 OCR 样本和 1 个英文 OCR 样本。
- 每个样本有 Ground Truth 文件。
- 样本可用于 CI 中的 OCR 准确率回归测试。

### 步骤 5：创建导出验证样本

**要求**：
- 为五种导出格式每种准备至少 1 个"期望输出"样本。
- 准备 1 个用于导出测试的源文档（Markdown 格式，含标题、段落、表格、图片、代码块）。

**源文档**：`samples/exports/source-test-document.md`

内容要求：
- 一级标题（文档标题）。
- 二级标题（章节标题）。
- 三级标题（子章节）。
- 多个段落（含中文、英文、数字）。
- 至少 1 个普通表格。
- 至少 1 个代码块。
- 至少 1 个图片引用。
- 分页标记（用于验证 PDF 分页）。
- 总页数不少于 3 页。

**期望输出目录**：`samples/exports/expected/`

每种格式放置一个"期望正确输出"文件，用于回归对比：
- `expected/pdf/expected-output.pdf`：由源文档正确导出后的 PDF。
- `expected/docx/expected-output.docx`：由源文档正确导出后的 DOCX。
- `expected/markdown/expected-output.md`：由源文档正确导出后的 Markdown。
- `expected/csv/expected-output.csv`：由源文档正确导出后的 CSV。
- `expected/txt/expected-output.txt`：由源文档正确导出后的 TXT。

**注意**：期望输出文件在导出功能实现后生成（第5-6周），现在先创建目录和源文档。

验收标准：
- 导出源文档包含所有必需元素（标题、段落、表格、代码块、图片、分页）。
- 期望输出目录结构就位。
- 导出功能实现后，期望输出文件被填充。

验收标准：
- 每种检索模式至少 5 个查询。
- 查询覆盖中文和英文。
- 查询覆盖正常匹配、边界匹配和无匹配场景。

### 步骤 7：编写样本验证脚本

**文件路径**：`scripts/validate-samples.sh`（或 Python 脚本）

**功能要求**：
1. 遍历 `samples/manifest.json` 中的所有样本。
2. 检查每个样本文件是否存在。
3. 检查文件大小是否与 manifest 中的记录一致。
4. 检查 SHA256 是否与 manifest 中的记录一致（如有）。
5. 输出验证报告：通过/失败的样本数量、缺失的文件列表、哈希不匹配的文件列表。

**使用方式**：
```bash
bash scripts/validate-samples.sh
# 或
uv run python scripts/validate_samples.py
```

**退出码**：
- 0：所有样本通过验证。
- 1：存在不通过的样本。

**集成到 CI**：
- 将此脚本添加到 CI 的 `quality` job 中，确保样本库完整性。

验收标准：
- 脚本能正确检测缺失文件、大小不一致和哈希不匹配。
- 脚本在 CI 中运行，样本库不完整时 CI 失败。

### 步骤 8：编写样本库维护文档

**文件路径**：`samples/README.md`（补充维护说明）

**追加内容**：
- 如何添加新格式样本：步骤说明。
- 如何更新样本：修改 manifest.json 和文件。
- 如何运行样本验证：命令和预期输出。
- 样本库版本管理：样本变更与 Git 提交绑定的策略。
- 禁止事项：不提交真实敏感数据、不提交超过 50MB 的单个文件。

验收标准：
- `samples/README.md` 包含完整的维护说明。
- 文档可由非作者（如员工4）独立操作。

## 五、协作要点

- 与员工4（文档处理后端）协作：确认每个格式的样本是否符合解析器预期输入格式。员工4 的解析器实现后，用样本库进行回归测试。
- 与员工5（RAG 与导出后端）协作：确认检索查询集和导出期望输出是否覆盖核心场景。员工5 的导出功能实现后，用期望输出文件进行对比验证。
- 与员工1（前端）协作：确认 Markdown 异常样本（含危险 HTML）能被前端 DOMPurify 正确过滤。
- 样本库是持续积累的：第2-3周建立基础，第4-8周根据测试发现不断补充。

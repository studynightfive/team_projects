## 四、执行步骤

### 步骤 1：创建 GitHub Actions CI 工作流

**文件路径**：`.github/workflows/ci.yml`

**工作流名称**：`CI`

**必需 Job 名称**：`quality`（PR 分支保护检查此名称）

**触发条件**：
- `push` 到 `main` 以外的所有分支
- `pull_request` 到 `main`

**Job 内容**（分阶段实现）：

**阶段 A（第1周，前端骨架出现前）**：
- 暂时只包含一个 pass-through 步骤，确保 CI 基础设施可用。
- 在 `quality` job 中输出构建环境信息（Node.js 版本、Python 版本、Docker 版本）。

**阶段 B（前端骨架出现后，追加以下步骤）**：
```text
npm ci
npm run typecheck:web
npm run lint:web
npm run test:web
npm run build:web
```

**阶段 C（后端骨架出现后，追加以下步骤）**：
```text
uv sync --project backend --frozen
uv run --project backend ruff check backend
uv run --project backend mypy backend/app
uv run --project backend pytest backend/tests
```

**关键要求**：
- `quality` job 中任何一个步骤失败，整个 job 失败，PR 无法合并。
- 使用 `actions/cache` 缓存 `node_modules` 和 Python 虚拟环境，加速后续构建。
- 使用 `actions/setup-node@v4` 和 `actions/setup-python@v5`，版本号精确匹配 `22.23.1` 和 `3.10.20`。
- 测试步骤使用 `pytest-cov` 生成覆盖率报告，覆盖率阈值暂设为 70%。

验收标准：
- 推送任意分支触发 CI 运行。
- 创建 PR 到 `main` 时，`quality` job 必须通过才能合并。
- CI 缓存生效，第二次运行时间明显短于第一次。

### 步骤 2：创建后端 API Server Dockerfile

**文件路径**：`deploy/docker/api-server.Dockerfile`

**要求**：
- 基础镜像使用 `python:3.10.20-slim`（精确版本）。
- 安装系统依赖：`libreoffice`、`tesseract-ocr`、`poppler-utils`、`ghostscript`、`libmagic`、`noto-cjk-fonts`，版本精确锁定。
- 使用 `uv` 安装 Python 依赖，执行 `uv sync --project backend --frozen`。
- 复制后端代码到容器内。
- 暴露端口 8000。
- 使用 `uvicorn` 启动，host 为 `0.0.0.0`，port 为 `8000`。
- 使用非 root 用户运行（创建 `appuser`）。
- 包含 `HEALTHCHECK` 指令，检查 `/api/v1/health/live`。

**注意**：
- 不要复制 `.env` 文件到镜像中。
- 不要硬编码任何 Secret。
- 所有 Secret 通过环境变量在运行时注入。

验收标准：
- `docker build -f deploy/docker/api-server.Dockerfile -t api-server:test .` 成功。
- 容器启动后 `/api/v1/health/live` 返回 200。
- 容器以非 root 用户运行（`docker exec <container> whoami` 不是 `root`）。

### 步骤 3：创建 Worker Dockerfile

**文件路径**：`deploy/docker/worker.Dockerfile`

**要求**：
- 基础镜像与 API Server 相同（`python:3.10.20-slim`），安装相同的系统依赖。
- 使用相同的 `uv sync` 安装依赖。
- 复制后端代码到容器内。
- 不暴露端口（Worker 不对外）。
- 使用 `arq` 启动 Worker。
- 使用非 root 用户运行。
- 包含 `HEALTHCHECK` 指令，检查 Worker 进程是否存活。

验收标准：
- `docker build -f deploy/docker/worker.Dockerfile -t worker:test .` 成功。
- 容器启动后 Worker 进程正常运行，能连接到 Redis。
- 容器以非 root 用户运行。

### 步骤 4：创建 Web Dockerfile 与 Nginx 配置

**文件路径**：
- `deploy/docker/web.Dockerfile`
- `deploy/nginx/default.conf`

**Web Dockerfile 要求**：
- 第一阶段（构建阶段）：使用 `node:22.23.1-alpine`，执行 `npm ci` 和 `npm run build:web`。
- 第二阶段（运行阶段）：使用 `nginx:1.28.0-alpine`。
- 从构建阶段复制 `frontend/web/dist/` 到 `/usr/share/nginx/html/`。
- 复制 Nginx 配置文件到 `/etc/nginx/conf.d/default.conf`。
- 暴露端口 80。

**Nginx 配置要求**：
- `/` 和 `/admin/*` 使用同一 SPA 产物（`/usr/share/nginx/html/`）。
- 未命中静态文件时 fallback 到 `/index.html`（SPA fallback）。
- `/api` 反向代理到 `api-server:8000`。
- SSE 代理配置：
  - `proxy_buffering off;`
  - `proxy_cache off;`
  - `proxy_read_timeout 300s;`
  - `proxy_set_header Connection '';`
  - `chunked_transfer_encoding on;`
- 上传大小限制 `client_max_body_size` 设置为合理值（如 100M）。
- 安全响应头：
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `X-XSS-Protection: 1; mode=block`
  - `Referrer-Policy: strict-origin-when-cross-origin`
- 生产环境额外配置（通过环境变量或模板控制）：
  - TLS 终止（如使用 Certbot 或外部 LB）
  - HSTS 头

**注意**：
- 开发环境（Vite dev server）不使用此 Nginx 配置，仅用于容器化部署。
- 前端开发时使用 Vite 的代理功能，配置见 `vite.config.ts`。

验收标准：
- `docker build -f deploy/docker/web.Dockerfile -t web:test .` 成功。
- 容器启动后访问 `/` 返回前端 SPA 页面。
- 访问 `/admin` 和 `/admin/users` 返回同一 SPA 页面（不是 404）。
- 访问 `/api/v1/health/live` 代理到后端。
- SSE 端点（`/api/v1/chat/stream`）代理缓冲已关闭。
- 上传 50M 文件不因 `client_max_body_size` 被拒绝。

### 步骤 5：创建 Docker Compose 编排

**文件路径**：
- `deploy/docker-compose.yml`（开发/联调环境）
- `deploy/docker-compose.test.yml`（测试环境）

**docker-compose.yml 要求**：

定义 5 个服务：`web`、`api-server`、`worker`、`postgres`、`redis`。

```yaml
services:
  web:
    build:
      context: ..
      dockerfile: deploy/docker/web.Dockerfile
    ports:
      - "80:80"
    depends_on:
      - api-server
    networks:
      - app-network

  api-server:
    build:
      context: ..
      dockerfile: deploy/docker/api-server.Dockerfile
    expose:
      - "8000"
    environment:
      - DATABASE_URL=postgresql+asyncpg://postgres:${POSTGRES_PASSWORD}@postgres:5432/knowledge_base
      - REDIS_URL=redis://redis:6379/0
      - SECRET_KEY=${SECRET_KEY}
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    volumes:
      - storage_data:/app/storage
    networks:
      - app-network

  worker:
    build:
      context: ..
      dockerfile: deploy/docker/worker.Dockerfile
    environment:
      - DATABASE_URL=postgresql+asyncpg://postgres:${POSTGRES_PASSWORD}@postgres:5432/knowledge_base
      - REDIS_URL=redis://redis:6379/0
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    volumes:
      - storage_data:/app/storage
    networks:
      - app-network

  postgres:
    image: pgvector/pgvector:pg17
    environment:
      - POSTGRES_DB=knowledge_base
      - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - app-network

  redis:
    image: redis:7.4.9-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - app-network

volumes:
  postgres_data:
  storage_data:

networks:
  app-network:
    driver: bridge
```

**docker-compose.test.yml 要求**：
- 与 `docker-compose.yml` 结构相同，但增加测试专用的环境变量（如 `TESTING=true`）。
- 不暴露端口（或使用不同端口，避免与开发环境冲突）。
- 测试完成后自动退出（`--abort-on-container-exit`）。

**关键要求**：
- 所有 Secret（密码、密钥）通过环境变量传入，不在 compose 文件中硬编码。
- 提供 `deploy/env/.env.example` 模板文件，列出所有必需的环境变量和说明。
- 提供 `deploy/env/.env.test.example` 测试环境变量模板。
- 健康检查覆盖所有服务：`web`（HTTP 200）、`api-server`（`/api/v1/health/live`）、`worker`（进程存活）、`postgres`（`pg_isready`）、`redis`（`redis-cli ping`）。
- 网络隔离：`postgres` 和 `redis` 不暴露到宿主机（仅内部网络），除非在开发环境中明确需要。

验收标准：
- `docker compose -f deploy/docker-compose.yml up -d` 所有5个服务启动成功。
- 所有服务健康检查通过。
- `docker compose -f deploy/docker-compose.yml down` 正确停止并清理。
- `docker compose -f deploy/docker-compose.test.yml up --build --abort-on-container-exit` 测试通过后自动退出。
- `deploy/env/.env.example` 包含所有必需环境变量和说明。

### 步骤 6：创建 Makefile 和脚本

**文件路径**：`Makefile`、`scripts/` 目录

**Makefile 要求**（提供常用命令的快捷入口）：

```makefile
.PHONY: dev build test clean

dev:
	# 启动开发环境
	docker compose -f deploy/docker-compose.yml up -d postgres redis

build:
	# 构建所有镜像
	docker compose -f deploy/docker-compose.yml build

test:
	# 运行完整容器测试
	docker compose -f deploy/docker-compose.test.yml up --build --abort-on-container-exit

clean:
	# 清理所有容器和卷
	docker compose -f deploy/docker-compose.yml down -v
	docker compose -f deploy/docker-compose.test.yml down -v
```

**scripts/ 目录要求**：
- `scripts/backup.sh`：数据库和文件备份脚本（第4个提示词中详细实现）。
- `scripts/restore.sh`：数据库和文件恢复脚本（第4个提示词中详细实现）。
- `scripts/health-check.sh`：全服务健康检查脚本。

验收标准：
- `make dev` 启动 postgres 和 redis 服务。
- `make build` 构建所有镜像。
- `make test` 运行完整测试。
- `make clean` 清理所有容器和卷。

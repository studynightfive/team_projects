### 步骤 2：建立不可变版本镜像构建流程

**镜像标签策略**：
- 每次发布使用两个标签：
  - 语义版本号：`v1.0.0`（递增）。
  - 提交 SHA：`abc1234`（Git 短 SHA）。
- 禁止生产使用 `latest` 标签。
- 禁止使用 `latest` 标签部署到任何非本地环境。

**构建脚本**：`scripts/build-release.sh`

```bash
#!/bin/bash
# 发布镜像构建脚本
# 用法: bash scripts/build-release.sh <version>

set -euo pipefail

VERSION="${1}"
COMMIT_SHA=$(git rev-parse --short HEAD)

echo "Building release images: version=${VERSION}, sha=${COMMIT_SHA}"

# 构建并标记镜像
docker build -f deploy/docker/api-server.Dockerfile \
  -t api-server:${VERSION} \
  -t api-server:${COMMIT_SHA} \
  .

docker build -f deploy/docker/worker.Dockerfile \
  -t worker:${VERSION} \
  -t worker:${COMMIT_SHA} \
  .

docker build -f deploy/docker/web.Dockerfile \
  -t web:${VERSION} \
  -t web:${COMMIT_SHA} \
  .

# 验证镜像
echo "Verifying images..."
docker run --rm api-server:${VERSION} python -c "print('API Server OK')"
docker run --rm worker:${VERSION} python -c "print('Worker OK')"

echo "Build complete: ${VERSION} (${COMMIT_SHA})"
```

**镜像验证要求**：
- 构建后立即验证镜像可启动。
- 验证健康检查端点可访问。
- 记录镜像 SHA256 到发布记录中。

**镜像存储**：
- 如果使用容器镜像仓库（如 Docker Hub、GitHub Container Registry、Harbor），推送镜像：
  ```bash
  docker tag api-server:${VERSION} registry.example.com/api-server:${VERSION}
  docker push registry.example.com/api-server:${VERSION}
  ```
- 如果使用本地镜像，通过 `docker save` 导出并归档。

验收标准：
- 发布镜像构建脚本可执行，生成带版本号和 SHA 标签的镜像。
- 构建后镜像验证通过。
- 镜像 SHA256 已记录。

### 步骤 3：建立生产部署流程

**文件路径**：`docs/operations/deployment.md`（更新或补充生产部署章节）

**生产部署步骤**：

**A. 部署前准备**：
1. 确认发布检查清单全部通过。
2. 确认备份已完成并验证。
3. 确认回滚镜像已准备（上一个稳定版本）。
4. 通知团队发布窗口和时间。
5. 准备发布记录（版本号、发布日期、变更内容）。

**B. 数据库迁移**：
1. 在维护窗口内，先执行数据库备份。
2. 执行数据库迁移：
   ```bash
   docker compose -f deploy/docker-compose.yml exec api-server \
     uv run alembic upgrade head
   ```
3. 验证迁移成功：检查关键表结构、测试基本查询。
4. 如果迁移涉及破坏性变更，确认已分阶段执行。

**C. 应用部署**：
1. 拉取或加载新版本镜像。
2. 更新 `docker-compose.yml` 中的镜像标签为发布版本。
3. 滚动更新服务：
   ```bash
   docker compose -f deploy/docker-compose.yml up -d --no-deps api-server worker web
   ```
4. 等待健康检查通过。
5. 验证核心功能：登录、检索、问答、导出。

**D. 部署后验证**：
1. 所有服务健康检查通过。
2. 登录、刷新和退出正常。
3. 知识库列表和文档预览正常。
4. 三种检索模式返回结果。
5. 流式问答正常，引用可定位。
6. 文档上传和转换正常。
7. 五种格式导出正常。
8. 管理功能正常（仅管理员）。
9. 监控指标正常采集。
10. 日志无异常错误。

验收标准：

### 步骤 4：建立回滚方案

**文件路径**：`docs/operations/rollback.md`

**回滚策略**：

**A. 应用镜像回滚**（优先执行，最快恢复）：
1. 将 `docker-compose.yml` 中的镜像标签改回上一个稳定版本。
2. 滚动更新服务：
   ```bash
   docker compose -f deploy/docker-compose.yml up -d --no-deps api-server worker web
   ```
3. 等待健康检查通过。
4. 验证核心功能恢复。

**B. 数据库回滚**（仅在必要时执行）：
- 优先回滚应用镜像，数据库不回滚。新版本应用的数据库迁移必须是向后兼容的。
- 如果必须回滚数据库：
  1. 从最近的备份恢复数据库（使用提示词04中的恢复脚本）。
  2. 验证数据完整性。
  3. 启动应用服务。

**C. 回滚决策树**：
```
发布后发现异常
├── 应用层问题（API 错误、前端异常）
│   └── 回滚应用镜像（5分钟内）
├── 数据库迁移问题（数据错误、性能下降）
│   ├── 迁移向后兼容？-> 回滚应用镜像，数据库不回滚
│   └── 迁移不兼容？-> 回滚应用镜像 + 恢复数据库备份
└── 数据损坏或丢失
    └── 回滚应用镜像 + 恢复数据库和文件备份
```

**D. 回滚演练**（第8周在测试环境执行）：

1. 部署新版本镜像到测试环境。
2. 模拟发布后发现问题（如注入一个已知错误）。
3. 执行回滚流程。
4. 验证回滚后服务正常。
5. 记录回滚耗时。

### 步骤 5：建立 Nginx 生产配置

**文件路径**：`deploy/nginx/default.conf`（补充生产环境配置）

**生产环境额外要求**（方案第17.3节）：

**A. TLS 配置**：
- 如果 Nginx 直接处理 TLS（不使用外部 LB）：
  ```nginx
  server {
      listen 443 ssl http2;
      ssl_certificate     /etc/nginx/certs/fullchain.pem;
      ssl_certificate_key /etc/nginx/certs/privkey.pem;
      ssl_protocols TLSv1.2 TLSv1.3;
      ssl_ciphers HIGH:!aNULL:!MD5;
  }
  ```
- 如果使用外部 LB 或 Certbot，提供对应的配置说明。

**B. 安全响应头**：
```nginx
add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options "DENY" always;
add_header X-XSS-Protection "1; mode=block" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

**C. HTTP 重定向到 HTTPS**：
```nginx
server {
    listen 80;
    return 301 https://$host$request_uri;
}
```

**D. 上传大小限制**：
```nginx
client_max_body_size 100m;
```

**注意**：
- 开发环境不使用 TLS（通过 `docker-compose.yml` 和 `docker-compose.test.yml` 区分）。
- 提供 `deploy/nginx/default.conf.template` 模板，通过环境变量控制 TLS 开关。

验收标准：
- Nginx 生产配置包含 TLS、安全响应头、HTTP 重定向和上传大小限制。
- 配置模板可通过环境变量切换开发/生产模式。
